# Account 집합체 도입 (available + locked)

사용자당 한 계좌를 사용 가능 / 잠금 두 bucket으로 나눠 들고, 의도 이름이 붙은 mutator로만 잔고를 바꾼다.

## 개요

`Account`는 시스템에서 유일하게 비관적 락(pessimistic lock) 아래에서 잔고가 바뀌는 entity다. 잔고를 `available`(쓸 수 있는 돈)과 `locked`(베팅 노출에 묶인 돈) 두 bucket으로 나눈 것이 이 repo의 핵심 모델 결정이다. 베팅이 접수되면 stake를 계좌 밖으로 빼지 않고 available에서 locked로 옮긴다. 돈이 계좌 안에 남아 있어야 운영 액션(노출 강제 환불)과 일일 reconciliation에서 추적이 된다.

한 계좌는 한 통화만 든다(one account = one currency). KRW와 USD를 둘 다 지원하지만 계좌 단위로는 단일 통화다.

## 작업 순서

1. `Money`(shared-protocol record)를 JPA가 받을 수 있는 `EmbeddedMoney` wrapper로 감싼다.
2. `Account` entity에 available / locked 두 `@Embedded` bucket과 `@Version`을 둔다.
3. bucket 전이를 의도 이름 mutator로 노출한다.
4. 각 mutator가 금액 양수 / 통화 일치 / 잔고 충분을 검증하게 한다.
5. 도메인 예외(`InsufficientBalanceException`, `CurrencyMismatchException`)를 둔다.

## 작업 내역

### `EmbeddedMoney.java`

Hibernate 6은 다른 모듈의 record(shared-protocol `Money`)를 `@Embeddable`로 직접 바인딩하지 못한다. 영속성 경계 전용 wrapper를 두되, 도메인 API는 여전히 `Money`를 주고받는다. C/C++이라면 struct를 그대로 재사용했을 자리에, ORM의 reflection 매핑 제약 때문에 어댑터 한 겹이 들어간다.

**책임:** `Money`(amount + currency)를 영속화 가능한 `@Embeddable`로 감싸고, 도메인 ↔ 영속성 변환 두 메서드를 노출한다.

### `Account.java`

mutator는 산술 이름(`add`)이 아니라 도메인 의도 이름을 쓴다. 호출부가 실제 자금 흐름으로 읽힌다. mutator 다섯 개가 자금 흐름에 1:1로 대응한다.

**책임:** available / locked 두 bucket과 `@Version`을 들고, 다섯 mutator로만 잔고를 바꾼다. 각 mutator는 *상태 변경 전에* 양수 / 통화 일치 / 잔고 충분을 검증하고, 산술은 overflow를 예외로 드러낸다(음수 잔고·통화 혼합이 entity 안으로 못 들어오게).

## 검증

- `mvn -q compile` — `Account` / `EmbeddedMoney`가 계약대로 컴파일되는지.
- 감각 점검(도메인 동작은 004 `AccountTest`가 고정): `moveAvailableToLocked`가 available 초과 시 `InsufficientBalanceException`을 던지는가 / 통화가 다른 `Money`에 `CurrencyMismatchException`인가 / `forfeitLocked` 뒤 available은 그대로이고 total만 줄어드는가.
