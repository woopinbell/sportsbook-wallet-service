# LedgerEntryRepository 읽기 메서드

append-only 분개장의 멱등성 조회와 계좌별 시간순 스캔을 파생 쿼리로 노출한다.

## 개요

`LedgerEntryRepository`는 append-only 분개장의 읽기 경로만 노출한다. Spring Data의 파생 쿼리(derived query)는 메서드 이름을 파싱해 쿼리를 생성한다. `findByIdempotencyKey`라는 이름만으로 `WHERE idempotency_key = ?`가 만들어진다. 이름이 곧 쿼리다.

## 작업 순서

1. `JpaRepository`를 상속한다.
2. 멱등성 조회 두 개(`findByIdempotencyKey`, `existsByIdempotencyKey`)를 둔다.
3. 계좌별 시간순 스캔(`findByAccountIdOrderByCreatedAtAsc`)을 둔다.

## 작업 내역

### `LedgerEntryRepository.java`

세 파생 쿼리가 각자 다른 경로를 받친다. 이름 규칙(`findBy…`, `existsBy…`, `…OrderBy…Asc`)으로 SQL 없이 쿼리를 만든다.

**책임:** 세 읽기를 *이름으로* 선언한다 — (1) 키에 대한 차변/대변 쌍(재시도 복구 handler가 이긴 쪽 응답 재구성), (2) 키 존재 probe(Redis 미스 시 가벼운 확인), (3) 계좌별 시간순 스캔(invariant listener / reconciliation). 변경(mutating) 메서드는 두지 않는다 — 분개장은 append-only이고 프로덕션은 `saveAll(pair)` 삽입만 한다.

```java
public interface LedgerEntryRepository extends JpaRepository<LedgerEntry, UUID> {
  // TODO: 키에 대한 차변/대변 쌍을 List로 (재시도 복구용)
  // TODO: 키 존재 여부 boolean probe (Redis 미스 fast path)
  // TODO: 계좌별 created_at 오름차순 스캔 (invariant / reconciliation)
}
```

## 검증

- `mvn -q compile` — 인터페이스가 컴파일되는지(메서드 이름이 파생 가능한 형태인지).
- 감각 점검(008 통합 테스트가 실제 DB에서 고정): `findByIdempotencyKey`가 한 키에 대해 차변+대변 *두* 행을 돌려주는가 / `findByAccountIdOrderByCreatedAtAsc`가 시간 순서를 보장하는가. 잘못된 이름(예: `findByIdempotency`)은 부팅 시 `PropertyReferenceException`으로 드러난다.
