# 결과 이벤트 발행 배선 (outbox 경유)

각 연산이 분개 쌍과 같은 트랜잭션에서 Avro 이벤트를 outbox에 쓰도록 배선하고, 실패 이벤트는 별도 트랜잭션에서 남긴다.

## 개요

`WalletService`가 이제 saga에 참여하는 각 연산마다 분개 쌍과 같은 DB 트랜잭션에서 `OutboxEvent`를 쓴다. 원자적 커밋이 핵심이다. publisher가 이후 어떻게 동작하든 이벤트가 원장 변경과 어긋날 수 없다.

## 작업 순서

1. `WalletEventFactory`로 Avro 이벤트 생성과 토픽/키 규약을 분리한다.
2. debit/credit 성공 시 같은 트랜잭션에서 outbox에 쓴다.
3. debit 실패 시 별도 트랜잭션에서 실패 이벤트를 남긴다.
4. `AvroSerializer`로 바이너리 인코딩을 캡슐화한다.

## 작업 내역

### 세 이벤트 형태

shared-protocol이 생성한 Avro 클래스를 쓴다(계약).

**책임:** 세 이벤트를 도메인 결과에 매핑한다 — `WalletDebited`(stake debit 성공), `WalletCredited`(reason enum이 `CreditCommand.Source`를 매핑: USER_LOCKED→REFUND, HOUSE_POOL→PAYOUT), `WalletDebitFailed`(예외 매핑: InsufficientBalance→INSUFFICIENT_BALANCE, AccountNotFound→ACCOUNT_NOT_FOUND, CurrencyMismatch→CURRENCY_MISMATCH). deposit과 withdraw는 V1에서 발행하지 않는다 — 베팅 saga가 보지 않는 운영성 연산이고 `WalletCredited`의 reason enum에 맞는 값이 없다.

### 성공 경로와 실패 경로의 트랜잭션 차이

성공 이벤트는 분개 쌍을 쓰는 `TransactionTemplate` 블록 *안에서* 쓰여 원장 커밋을 공유한다. 실패 이벤트는 별도 fresh 트랜잭션(`publishDebitFailure`)에서 쓴다. 원래 트랜잭션이 롤백되므로 거기서 쓰면 이벤트도 같이 사라지기 때문이다.

**책임:** 아래 catch에서 실패 이벤트를 별도 트랜잭션에 남기고 트리거한 예외는 caller가 다시 던지게 한다. 별도 트랜잭션이 허용되는 이유: 실패가 hot path가 아니고(전체 debit 중 소수여야 함), 대안("롤백 밖으로 side-channel payload를 빼기")이 entity manager를 복잡하게 만든다.

```java
} catch (InsufficientBalanceException e) {
  // TODO: 이 예외에 대응하는 실패 이벤트를 별도 트랜잭션에 남긴 뒤, 트리거한 예외는 caller가 다시 던지게 한다
}
```

### `WalletEventFactory`와 토픽 규약

토픽/스키마 상수는 producer/consumer가 공유하는 계약이라 노출한다. 모든 wallet 이벤트가 userId를 파티션 키로 공유해 토픽에서 사용자별 순서가 보존된다(ADR-0006).

**책임:** 각 factory 메서드가 Avro 레코드를 빌드해 `(topic, schema, partitionKey=userId, payload)`로 `OutboxEvent`를 만든다. `AvroSerializer`는 Schema Registry 없이 바이너리 인코딩하는 helper다 — 스키마 id는 양쪽 shared-protocol `SpecificRecord`에 암묵적(ADR-0014).

```java
public class WalletEventFactory {
  static final String TOPIC_DEBITED = "wallet.debited.v1";
  static final String TOPIC_CREDITED = "wallet.credited.v1";
  static final String TOPIC_DEBIT_FAILED = "wallet.debit-failed.v1";
  // SCHEMA_DEBITED / SCHEMA_CREDITED / SCHEMA_DEBIT_FAILED = Avro 레코드 이름

  public OutboxEvent debited(/* ... */) {
    // TODO: WalletDebited Avro 빌드 → wrap(TOPIC_DEBITED, SCHEMA_DEBITED, partitionKey=userId, ...)
  }
  // credited(...) / debitFailed(...) 동형
}
```

### Avro logical type 마찰

`occurredAt`은 `timestamp-millis` logical type인데 shared-protocol이 jsr310으로 생성돼 setter가 `long`이 아니라 `Instant`를 받는다. `now.toEpochMilli()`를 넘기면 컴파일 에러가 난다 — `now`(Instant)를 그대로 넘긴다.

## 검증

- `mvn -q -Dtest=OutboxIntegrationTest test` — 성공 이벤트가 분개와 같은 트랜잭션에, 실패 이벤트가 메인 롤백에도 남는지(012의 1~4번). *Testcontainers + embedded Kafka가 떠야 한다.*
- 감각 점검: InsufficientBalance debit 후 메인 트랜잭션은 롤백됐는데 `WalletDebitFailed` 행은 남는가(별도 트랜잭션 증거) / deposit·withdraw가 이벤트를 만들지 *않는가* / HOUSE_POOL credit이 reason=PAYOUT, USER_LOCKED credit이 reason=REFUND인가.
