# 문제지 사용 가이드

## 목적

`docs/practice/`는 `docs/commits/`(답지)에서 파생한 문제지다. 각 `NNN.md`는 같은 번호 답지의 제목·개요·작업 순서·논리 단위 제목을 그대로 두고, 코드 본문을 `// TODO: <책임>` 골격으로 비운 것이다. 답을 보지 않고 각 단위를 직접 구현하며 `wallet-service`를 도메인 모델부터 부하 증명까지 쌓아 올리는 용도다.

double-entry ledger / 멱등성 3층 / transactional outbox / 정합성 이중화처럼 이 저장소가 강조하는 결정을, 답을 베끼지 않고 *재유도*하는 데 초점을 둔다.

## 사용법

1. `001.md`부터 번호 순서대로 진행한다.
2. 각 문서에서 `## 개요`와 `## 작업 순서`로 무엇을 만들지 잡는다.
3. `### 논리 단위`마다 `**책임:**` 한 줄을 읽고, 노출된 시그니처/계약 위에서 `// TODO`를 실제 구현으로 채운다.
4. 막히면 같은 번호의 답지 `../commits/NNN.md`를 펼친다.
5. 각 문서 끝 `## 검증`의 명령으로 통과 여부를 확인한다.

## 코드 노출 수준

답지의 `## 기억/설명 Level` 태그에 따라 비우는 정도가 다르다. 판정 한 줄은 **"비우면 시작점을 잃나, 풀 문제가 생기나?"** — 시작점 손실은 노출, 문제 발생은 비움이다.

- **재유도 대상(L3/L2)은 비운다**: 함수 본문, 알고리즘·제어흐름(예: `007`의 `runIdempotent` 3층, `010`의 publisher 루프, `016`의 invariant 네 검사, `020`의 forfeit leg).
- **계약은 노출한다**: 함수 시그니처, 구조체/record 이름, enum, 어노테이션, 공개 API 경로 표, 자명한 보일러플레이트. 학습자가 시작점을 잃지 않게.
- **스키마(`003`/`009`)는 얇게 비운다**: 컬럼 골격(매핑 계약)은 노출하고, CHECK·복합 unique·partial index 같은 *제약/인덱스 설계*만 비운다.
- **config/test는 구조 노출·값 비움**: k6 threshold 값(`019`), producer 설정 값(`010`), 테스트 단언/입력(`004`/`008`/`012`/`015`/`018`).

## 문서 범위

`001`–`020`이 실체 있는 dev 커밋(+ Phase 5 통합 `020`)에 1:1 대응한다. 답지의 회고 섹션(`## 결과`·`## 요약`·`## 다음 작업`·`## 핵심 확인`·`## 기억/설명 Level`)은 문제지에서 제외했다.

빠진 번호:

- **`000` (chore: 골격 초기화)** — 문제지 없음. pom 의존성·BOM 상속·빈 진입점은 *재유도할 설계가 아니라 템플릿 채우기*다(답지의 Level이 전부 L1). 답지만 둔다.
- **측정 수치(`019`의 dev-host 표)** — 호스트 의존 관찰 결과라 재유도 대상이 아니다. 직접 돌려 자기 수치를 답지와 비교하는 단위로 남겼다.

| 문서 구간 | 중심 질문 |
| --- | --- |
| `001`–`002` | 잔고를 available/locked로 나누고, 모든 이동을 차변/대변 쌍으로 기록하는 모델을 어떻게 세울 것인가? |
| `003` | 그 모델의 invariant(통화·음수·matched-pair 멱등성)를 SQL 제약으로 어떻게 박을 것인가? |
| `004` | 도메인 산술·`pair()`·UUID v7을 어떤 단위 테스트로 고정할 것인가? |
| `005`–`006` | 비관적 락 조회와 멱등성/스캔 읽기를 repository에 어떻게 나눌 것인가? |
| `007` | 네 연산이 공유하는 멱등 전송 primitive(Redis / 비관적 락 / DB 제약 3층)를 어떻게 만들 것인가? |
| `008` | 100건 동시 debit이 단일 분개 쌍으로 수렴함을 실제 DB 위에서 어떻게 증명할 것인가? |
| `009`–`012` | 잔고 변경과 같은 트랜잭션에서 이벤트를 쌓고 at-least-once로 흘리는 outbox를 어떻게 배선·검증할 것인가? |

| `016`–`018` | per-operation 검사와 일일 배치로 ledger 정합성을 어떻게 이중으로 지키고 drift를 어떻게 시험할 것인가? |
| `019` | k6 하니스로 처리량과 Level 3 정합성을 더 큰 규모에서 어떻게 증명할 것인가? |
| `020` | 정산 4결과(win/push/void/lose) 중 빠진 LOST leg를 기존 primitive 재사용으로 어떻게 채울 것인가? |

## 전체 검증

대부분 문서의 `## 검증`은 `mvn -q -Dtest=<TestClass> test`다. 통합 테스트(`008`/`012`/`016`/`017`/`018`/`020` 일부)는 **Testcontainers(Docker)** 가, `012`는 추가로 embedded Kafka가, `019`는 docker-compose 부하 스택이 떠야 한다. 순수 단위(`004`)와 웹 슬라이스(`013`/`014`/`015`)는 Docker 없이 돈다. shared-protocol은 mavenLocal 의존이라 wallet 빌드 전에 그쪽에서 `mvn install`이 선행돼야 한다(답지 `000` 참조).
