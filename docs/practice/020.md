# 패배 스테이크 forfeit (locked → house)

베팅이 졌을 때 user의 locked 스테이크를 HOUSE로 capture하는 멱등 연산을 추가해 LOST 정산의 빠진 wallet leg를 채운다 (Phase 5 통합).

## 개요

이 커밋은 dev 단계가 아니라 **Phase 5 전체 스택 통합(orchestration e2e)에서 드러난 결함**의 wallet 몫이다.

settlement의 정산 결과는 WON / LOST / PUSH / VOID 네 가지인데, wallet에는 그중 **LOST에 대응하는 연산이 없었다**. 이긴 베팅은 스테이크를 `BET_REFUND`로 locked→available 되돌리고 수익을 `BET_PAYOUT`으로 지급한다. push/void도 `BET_REFUND`로 환불한다. 그런데 **진 베팅은** 묶인 스테이크를 회수하는 leg가 없어, 사용자가 잃은 돈이 영영 user의 locked bucket에 묶여 있었다.

도메인에는 이미 `Account.forfeitLocked` mutator와 `HOUSE` 시스템 계좌가 있었지만 **어떤 연산에도 배선돼 있지 않았다**. 이 커밋이 그 자리를 채운다.

## 작업 순서

1. `BET_FORFEIT` ledger reason을 추가한다(`LedgerReason` enum).
2. `ledger_entry_reason_valid` CHECK 제약을 V3 마이그레이션으로 확장한다.
3. `WalletService.forfeit`를 기존 멱등 primitive(`runIdempotent` + `writePair`) 위에 올린다.
4. `ForfeitCommand`와 `POST /internal/v1/wallet/transactions/forfeit` endpoint를 추가한다.
5. 통합 테스트로 locked→house capture, over-locked 거절, 합 = 0 invariant를 고정한다.

## 작업 내역

### 정산 매트릭스를 완성하는 leg

forfeit는 새 멱등성·락 메커니즘을 만들지 않는다. [007](../commits/007.md)이 세운 `runIdempotent` + `writePair` primitive를 그대로 재사용하고, leg 매핑과 reason만 추가한다. 정산 4결과를 나란히 두면 빈칸이 보인다.

```
win  : DEBIT user.AVAILABLE  ← CREDIT user.LOCKED      (BET_REFUND  · 스테이크 반환)
     + DEBIT user.AVAILABLE  ← CREDIT HOUSE.AVAILABLE  (BET_PAYOUT  · 수익 지급)
push : DEBIT user.AVAILABLE  ← CREDIT user.LOCKED      (BET_REFUND)
void : DEBIT user.AVAILABLE  ← CREDIT user.LOCKED      (BET_REFUND)
lose : ?                      ← ?                       (?)          ← 이 커밋이 채울 빈칸
```

**책임:** `forfeit`를 `runIdempotent`로 감싸 사용자 계좌에 비관적 락을 걸고 `account.forfeitLocked(amount, now)`로 잔고를 바꾼 뒤 `writePair`로 위 `lose` 행의 쌍을 쓴다. 다른 연산과 달리 **business-rejection 분기가 없다** — locked bucket이 접수 시 이미 스테이크를 들고 있어 정상 흐름에선 잔고 부족이 안 난다. 잘못된 호출(locked보다 큰 금액)만 도메인 mutator가 `InsufficientBalanceException`으로 막는다.

### V3 마이그레이션 — CHECK 제약 확장

V1의 `ledger_entry_reason_valid` CHECK는 다섯 reason만 허용했다. 새 `BET_FORFEIT` 행은 이 제약에 걸려 거부되므로, 제약을 DROP한 뒤 reason 하나를 더해 다시 ADD한다. 기존 행은 모두 원래 다섯 reason이라 재검증은 no-op이다.

### 멱등성 — settlement의 재전달에 안전

**책임:** settlement은 LOST 베팅마다 `settle:forfeit:{betId}` 키로 이 endpoint를 부른다. Kafka 재전달이나 admin replay로 같은 LOST가 두 번 와도, [007](../commits/007.md)의 멱등성 3층이 두 번째 호출을 원래 결과로 흡수해 **double-capture(스테이크 이중 회수)가 일어나지 않는다**. forfeit는 새 키를 쓸 뿐, 멱등성 보장 자체는 기존 primitive가 한다.

`ForfeitCommand(userId, amount, idempotencyKey)`와 `POST /internal/v1/wallet/transactions/forfeit` endpoint(다른 transaction endpoint와 동형, `Idempotency-Key` 헤더)를 추가한다.
