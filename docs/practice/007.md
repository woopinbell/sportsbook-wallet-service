# WalletService 멱등 전송 primitive

deposit / withdraw / debit / credit 네 연산이 공유하는 멱등 전송 primitive를 세 층의 멱등성 보장과 함께 만든다.

## 개요

`WalletService`는 쓰기 측 핵심 애플리케이션 서비스다. V1의 네 연산이 하나의 전송 primitive를 공유하고, 그 primitive가 멱등성 계약(idempotency contract)을 끝까지 책임진다. 이 커밋이 이 repo에서 가장 밀도 높은 결정 묶음이다. 비관적 락, 멱등성 3층, 트랜잭션 경계 선택이 한자리에 모인다.

## 작업 순서

1. Redis fast path 캐시(`IdempotencyCache`)를 둔다.
2. `runIdempotent` 골격에 fast path / write path / race-loser path 세 층을 짠다.
3. 네 연산을 transfer leg 매핑으로 primitive 위에 올린다.
4. 트랜잭션 경계를 `TransactionTemplate`(프로그래밍 방식)으로 잡는다.
5. `Clock`을 주입해 시각을 테스트에서 고정 가능하게 한다.

## 작업 내역

### 멱등성 3층 (ADR-0005)

DB `UNIQUE(idempotency_key, side)`가 멱등성의 *강한* 보장이고 Redis는 가속(fast path)일 뿐이다. 세 층의 역할이 다르다.

**책임:**
1. **Fast path** — `IdempotencyCache`(Redis `StringRedisTemplate`, 24h TTL, prefix `idempotency:wallet:`)가 키를 본 적 있으면 그 그룹의 분개 쌍을 DB에서 가져와 원래 `WalletOperationResult`로 재구성한다.
2. **Write path** — `TransactionTemplate`으로 트랜잭션을 열고 `findByUserIdForUpdate`로 행을 락 건 뒤 도메인 mutator로 잔고를 바꾸고 `pair()`로 차변/대변을 삽입, 커밋 후 그룹 ID를 Redis에 캐싱.
3. **Race-loser path** — 같은 키 동시 요청 중 하나가 이겨 `UNIQUE(idempotency_key, side)`가 터지면, 진 트랜잭션은 롤백되고 catch가 이긴 쪽 쌍을 새 트랜잭션에서 다시 읽는다. 재시도 payload(userId/amount/reason)를 원본과 대조해 불일치면 `IdempotencyConflictException`(→ 409), 동일하면 같은 결과.

```java
private WalletOperationResult runIdempotent(IdempotencyKey key, UUID userId, Money amount,
    LedgerReason expectedReason, Supplier<WalletOperationResult> action) {
  // TODO 1) fast path: cache.lookupOperationGroup(key) 있으면 loadExisting + validateRetryMatches → 반환
  // TODO 2) write path: writeTx.execute(s -> action.get()) 성공 → cache.markProcessed → 반환
  // TODO 3) catch (DataIntegrityViolationException): 이긴 쪽 loadExisting,
  //         비었으면 rethrow / 있으면 validateRetryMatches 후 그 결과 반환
}
```

### 왜 `@Transactional`이 아니라 `TransactionTemplate`인가

race-loser path는 `DataIntegrityViolationException`을 잡은 뒤 이긴 쪽 쌍을 *다시 읽어야* 한다. 메서드 레벨 `@Transactional` 안에서 unique 위반이 나면 그 트랜잭션은 rollback-only로 표시되고, 같은 트랜잭션에서 추가 read를 하면 `TransactionSystemException`이 난다. 프로그래밍 방식 경계(`TransactionTemplate.execute`)는 catch를 실패한 write 트랜잭션 *바깥에서* 돌게 해 이 문제를 피한다. 선언적 트랜잭션을 "그냥 붙이면 되는 것"으로 보면 놓치는 함정이다.

### 연산별 leg 매핑

각 연산은 적절한 (목적지=차변, 출발지=대변) leg로 `pair()`를 호출할 뿐이다. 멱등성과 락은 primitive가 처리한다.

**책임:** 아래 표의 빈 칸을 채운다. 하나(debit)만 예시로 노출한다.

```
deposit:              DEBIT user.AVAILABLE      ← CREDIT EXTERNAL_PAYMENT.AVAILABLE   (예시)
withdraw:             ?                          ← ?
debit:                ?                          ← ?                       reason = ?
credit (USER_LOCKED): ?                          ← ?                       reason = ?
credit (HOUSE_POOL):  ?                          ← ?                       reason = ?
```

`openAccount`는 유일하게 호출자 멱등성 키가 없는 연산이다. userId 자체가 자연 키이고, PK 경합은 삽입 시 `DataIntegrityViolation`을 잡고 다시 읽어 처리한다.

명령은 record로 받는다(계약): `OpenAccountCommand(userId, currency)`, `DepositCommand` / `WithdrawCommand` / `DebitCommand(userId, amount, idempotencyKey)`, `CreditCommand(userId, amount, Source, idempotencyKey)` — `Source { USER_LOCKED, HOUSE_POOL }`. 결과는 `WalletOperationResult(operationGroupId, …)`.

### `Clock` 주입

`WalletInfrastructureConfig`가 `Clock`(UTC system clock)과 `TransactionTemplate`을 bean으로 wire한다.

**책임:** `WalletService`가 `Instant.now()`를 직접 부르지 않고 `Clock`에 의존하게 해, 테스트가 JVM 상태를 건드리지 않고 시각을 고정할 수 있게 한다.

## 검증

- `mvn -q -Dtest=WalletServiceIntegrationTest test` — 멱등성·동시성·거부 경로가 실제 PostgreSQL+Redis 위에서 green(008). *Testcontainers(Docker)가 떠야 한다.*
- 감각 점검: 같은 키 재시도가 부수효과를 *다시 실행하지 않고* 같은 `operation_group_id`를 돌려주는가 / 같은 키·다른 금액이 `IdempotencyConflictException`인가 / write path를 `@Transactional` 메서드로 바꾸면 race-loser의 재read가 `TransactionSystemException`으로 깨지는가.
