# 도메인 단위 테스트 (Account / LedgerEntry / UuidV7)

Spring 컨텍스트 없이 도는 순수 JUnit 테스트로 도메인 산술과 분개 쌍, UUID 비트 레이아웃을 덮는다.

## 개요

세 테스트 클래스 모두 Spring 컨텍스트 없이 돈다. 도메인이 결정적(deterministic)이라 수십 ms 안에 끝나고, 이후 통합 테스트가 의존하는 안쪽 루프가 된다. JUnit5 + AssertJ 조합을 쓴다.

## 작업 순서

1. `AccountTest`를 bucket 전이별 4개 nested 그룹으로 나눈다.
2. `LedgerEntryTest`로 `pair()` 팩토리의 invariant를 검증한다.
3. `UuidV7Test`로 RFC 9562 비트 레이아웃과 시간 정렬을 검증한다.

## 작업 내역

> 케이스 의도와 이름은 노출한다. 입력 fixture와 assertion 본문은 비운다 — 거기서 실제로 손이 움직인다.

### `AccountTest`

`@Nested`로 mutator별 그룹을 나눈다. JUnit5의 nested 클래스는 테스트를 도메인 동작 단위로 묶어 실패 위치를 좁힌다.

**책임:** 네 그룹 = staking(available→locked) / locked 반환 / 입출금(increase·decrease) / 패배 몰수(forfeit). 각 그룹이 happy path와 overdraft / 통화 불일치 / 음수 입력 거부를 함께 본다. forfeit는 locked만 줄고 available은 그대로이며 total이 몰수액만큼 감소함을 본다.

```java

  @Test void moves_funds_and_keeps_total_invariant() {
    // TODO: stake 이동 후 available↓ / locked↑ / total 불변
  }
  @Test void rejects_when_amount_exceeds_available() {
    // TODO: 가용 초과 stake → InsufficientBalanceException, requested/available 값까지 검증
  }
}
```

### `LedgerEntryTest`

`pair()` 팩토리는 모든 service 연산이 거치는 단일 통로라 invariant를 명시적으로 검증한다.

**책임:** 차변/대변이 키·그룹 ID는 공유하되 entry_id는 각자 다름을 고정한다. 비양수 금액 거부와 `TransferLeg` compact 생성자의 null 거부도 본다.

```java
// TODO: pair.debit().idempotencyKey() == pair.credit().idempotencyKey()
// TODO: pair.debit().operationGroupId() == pair.credit().operationGroupId()
// TODO: pair.debit().entryId() != pair.credit().entryId()
```

### `UuidV7Test`

**책임:** version/variant 비트와 시간 정렬을 본다. timestamp 주입 오버로드로 오름차순 입력이 오름차순 48비트 prefix를 만드는지(=B-tree 친화 삽입의 근거), 1000번 연속 생성이 모두 distinct한지 확인한다.

```java
// TODO: id.version() == 7
// TODO: id.variant() == 2   (RFC 9562 variant 10 → java.util.UUID#variant() == 2)
```

## 검증

- `mvn -q -Dtest=AccountTest,LedgerEntryTest,UuidV7Test test` — 세 클래스가 모두 green. 순수 단위 테스트라 컨테이너/Docker가 필요 없고 수십 ms에 끝난다.
- 감각 점검: 비운 그룹/assertion이 채워지기 전에는 빨강이어야 한다(테스트가 실제로 도메인 동작을 잡고 있다는 신호).
