# Flyway V2 스키마 (outbox_event)

잔고 변경과 같은 커밋 안에서 Kafka 이벤트를 쌓을 transactional outbox 테이블을 만든다.

## 개요

transactional outbox는 분산 트랜잭션의 부분 실패를 피하는 패턴이다(ADR-0006). DB 트랜잭션 커밋과 메시지 발행이 따로 일어나면, 커밋은 됐는데 발행이 실패하거나 그 반대가 생긴다. outbox는 이벤트를 잔고 변경과 같은 트랜잭션 안에서 테이블에 쓰고, 별도 publisher가 그 테이블을 읽어 Kafka로 보낸다. 커밋이 원자적이므로 이벤트가 원장 변경과 어긋날 수 없다.

이 커밋은 그 테이블만 만든다. entity와 publisher는 다음 커밋이다.

## 작업 순서

1. `outbox_event` 테이블에 이벤트 식별 / 발행 상태 컬럼을 둔다.
2. 미발행 행을 빠르게 찾는 partial index를 만든다.

## 작업 내역

### `outbox_event` 테이블

컬럼은 다음 커밋의 entity가 매핑할 계약이라 노출한다.

```sql
CREATE TABLE outbox_event (
    event_id      UUID                     PRIMARY KEY,
    topic         VARCHAR(64)              NOT NULL,
    partition_key VARCHAR(64)              NOT NULL,  -- 발행 시 Kafka 키. userId 문자열 → 사용자별 파티션 순서(ADR-0006)
    schema_name   VARCHAR(64)              NOT NULL,  -- Avro 레코드 이름. 발행엔 불필요, 진단용
    payload       BYTEA                    NOT NULL,  -- Avro 바이너리. Schema Registry 없음(ADR-0014)
    created_at    TIMESTAMP WITH TIME ZONE NOT NULL,
    published_at  TIMESTAMP WITH TIME ZONE            -- ack 전까지 NULL. 발행돼도 행은 안 지움(replay)
);
```

### partial index

partial index는 조건에 맞는 행만 인덱싱하는 PostgreSQL 기능이다.

**책임:** "미발행 행을 오래된 순으로"가 hot read path다. 대부분의 행은 곧 발행돼 인덱스에서 빠지므로 인덱스를 작게 유지하려면 — 어떤 컬럼 순서로, 어떤 조건(`WHERE`)으로 인덱싱해야 하나?

```sql
CREATE INDEX ix_outbox_unpublished ON outbox_event (/* TODO: 정렬 컬럼 */) WHERE /* TODO: 미발행 조건 */;
```

## 검증

- `mvn -q -Dtest=OutboxIntegrationTest test` — 통합 테스트(012)가 부팅하며 Flyway가 V2를 실제 PostgreSQL에 적용한다. *Testcontainers(Docker)가 떠야 한다.*
- 감각 점검: publisher(010/012)가 미발행 행을 가져올 때 partial index가 `published_at IS NULL` 조건을 받치는가(`EXPLAIN`으로 index scan 확인) / 발행 후에도 행이 테이블에 남는가(지우지 않음).
