# LedgerEntry 분개장 도입 (double-entry)

모든 자금 이동을 차변/대변 쌍으로 기록하는 append-only 분개 entry와 멱등성 제약 모델을 정의한다.

## 개요

`LedgerEntry`는 double-entry ledger의 불변 building block이다. 모든 자금 이동은 `LedgerEntry.pair()`가 만드는 차변(debit)/대변(credit) 쌍으로 기록된다. 두 entry는 같은 `operation_group_id`와 `idempotency_key`를 공유하고 `side`로만 갈린다. 차변은 목적지 (계좌, bucket)에, 대변은 출발지에 붙는다.

이 repo가 쓰는 규약은 차변 = bucket 증가, 대변 = bucket 감소다. 시스템 전체에서 차변 합에서 대변 합을 뺀 값은 항상 0이다. 이 invariant가 이후 after-commit listener와 일일 reconciliation이 검증하는 대상이다.

## 작업 순서

1. side / bucket / reason / 시스템 계좌 ID를 작은 enum과 상수로 박는다.
2. UUID v7 생성기를 둔다(시간 정렬 PK).
3. `LedgerEntry` entity와 `pair()` 팩토리를 만든다.
4. 멱등성 제약을 `UNIQUE(idempotency_key, side)`로 설계한다.

## 작업 내역

### 멱등성의 matched-pair 충돌

ADR-0005는 `idempotency_key`에 unique 제약을 걸라고 한다. 그런데 double-entry는 한 연산이 같은 키를 가진 두 행을 만든다. 단순 `UNIQUE(idempotency_key)`는 쌍의 두 번째 행을 거부한다. 이 충돌은 ADR을 읽을 때는 안 보이고 double-entry 구조와 만나서야 드러난다.

**책임:** 키당 차변 1 + 대변 1을 허용하면서 같은 키 재시도는 아무 행도 더 만들지 못하게 하는 제약 튜플을 설계한다(어떤 컬럼을 키에 더해야 쌍이 양립하는가?). 실제 SQL 제약은 V1 스키마(003)에서 물질화된다.

### enum과 시스템 계좌

enum 셋은 분개의 계약이라 노출한다. `SystemAccountIds`의 고정 UUID 구성은 비운다.

```java
enum BalanceBucket { AVAILABLE, LOCKED }
enum LedgerSide { DEBIT, CREDIT }
enum LedgerReason { DEPOSIT, WITHDRAW, BET_DEBIT, BET_PAYOUT, BET_REFUND }
```

**책임:** `HOUSE`·`EXTERNAL_PAYMENT`는 외부 세계가 차변/대변을 걸 상대다. account 테이블에 행이 없는 고정 UUID를 줘서, 외부 상태를 모델링하지 않고도 합 = 0 invariant가 성립하게 한다.

## 검증

- `mvn -q compile` — entity / record / enum / 생성기가 계약대로 컴파일되는지.
- 감각 점검(004 `LedgerEntryTest` / `UuidV7Test`가 고정): `pair()`의 차변·대변이 `idempotencyKey`·`operationGroupId`는 같고 `entryId`만 다른가 / 비양수 금액이 생성 시점에 거부되는가 / `generate()`의 `version()==7`·`variant()==2`이고 오름차순 timestamp가 오름차순 48비트 prefix를 만드는가.
