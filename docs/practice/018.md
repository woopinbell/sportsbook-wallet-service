# 정합성 테스트 (invariant 리스너 + reconciliation drift)

인위적 drift를 주입해 invariant 리스너와 reconciliation 배치가 잡아내는지 검증한다.

## 개요

`IntegrityIntegrationTest`가 Testcontainers PostgreSQL + Redis 위에서 두 정합성 장치를 검증한다. 고의로 깨진 데이터(짝 없는 차변)를 주입하고 Micrometer 카운터가 오르는지 본다. Kafka는 제외한다 — 리스너도 배치도 Kafka를 건드리지 않고, 빼면 몇 초 절약된다.

## 작업 순서

1. 정상 debit이 검사 카운터만 올리고 위반은 안 올림을 본다.
2. 짝 없는 차변을 주입하고 `InvariantChecker.verify()`를 직접 호출한다.
3. 같은 drift를 둔 채 `reconcile()`을 돌려 위반 카운터를 본다.

## 작업 내역

### 세 테스트

케이스 의도만 노출하고 단언은 비운다.

1. 정상 debit이 `wallet.invariant.checks`를 올리되 `wallet.invariant.violations`는 기준선 그대로. `@TransactionalEventListener(AFTER_COMMIT)` 배선을 end-to-end로 탄다(`debit()`이 `writePair()`에서 publish, 리스너가 커밋 후 fire).
2. 짝 없는 차변이 `InvariantChecker`를 트립한다.
3. 같은 drift로 `reconcile()`을 돌리면 위반 *두 개*(시스템 net != 0 하나, 계좌별 스냅샷 대조 하나).

### 직접 호출의 이유

이 결정은 실제로 한 번 빨갛게 뜬 뒤에 내렸다. 처음엔 `publishEvent`로 합성 이벤트를 쐈는데 AFTER_COMMIT 리스너가 감싸는 트랜잭션이 없으면 디스패치 자체를 안 해서 카운터가 0이었다.

**책임:** AFTER_COMMIT 배선은 위 1번(정상 debit) 테스트가 이미 증명한다. 합성 그룹은 감싸는 트랜잭션이 없어 리스너가 디스패치하지 않으므로, drift 테스트는 검증 로직을 *직접* 부른다.

### 합성 행을 스키마 적합하게

## 검증

- `mvn -q -Dtest=IntegrityIntegrationTest test` — 세 테스트 green. *Testcontainers(PostgreSQL+Redis)가 떠야 한다.*
- 감각 점검: 합성 이벤트를 `publishEvent`로 쏘면(직접 호출 대신) 리스너가 디스패치되지 않아 카운터가 0인 걸 재현할 수 있는가(그래서 직접 호출이 필요) / 한 drift가 reconciliation에서 두 위반으로 집계되는가.
