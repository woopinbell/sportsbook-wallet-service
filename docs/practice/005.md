# AccountRepository 비관적 락 조회

잔고를 바꾸는 모든 경로가 거치는 `SELECT FOR UPDATE` 조회를 repository에 노출한다.

## 개요

Spring Data JPA는 인터페이스만 선언하면 구현을 런타임에 생성하는 repository 추상화다. 메서드 시그니처와 어노테이션만으로 쿼리가 만들어진다. `AccountRepository`는 두 가지 읽기 경로를 구분해 노출한다. 락 없는 조회와 락 거는 조회다.

## 작업 순서

1. `JpaRepository<Account, UUID>`를 상속해 기본 CRUD를 얻는다.
2. `findByUserIdForUpdate`에 `@Lock(PESSIMISTIC_WRITE)`를 걸어 `SELECT FOR UPDATE`로 만든다.
3. 동반 예외 `AccountNotFoundException`을 둔다.

## 작업 내역

### `AccountRepository.java`

두 경로의 의미가 다르다. `findById(userId)`(JpaRepository 상속, 락 없음)는 잔고 조회 endpoint와 after-commit invariant listener가 쓰고, 락 거는 조회는 잔고를 바꾸는 연산만 쓴다.

**책임:** 사용자 계좌 행에 `SELECT ... FOR UPDATE`를 거는 조회를 노출한다. 락이 행을 잡은 채 같은 트랜잭션에서 ledger entry가 들어가고, 이것이 같은 사용자에 대한 동시 debit을 직렬화한다(ADR-0005). 메서드 이름이 호출부 의도로 읽히도록 이름 파생 대신 명시 조합을 쓴다.

```java
public interface AccountRepository extends JpaRepository<Account, UUID> {
  // TODO: 행을 락 걸고 가져오는 조회 — @Lock(PESSIMISTIC_WRITE) + @Query 명시 조합
  Optional<Account> findByUserIdForUpdate(UUID userId);
}
```

### `AccountNotFoundException`

`com.sportsbook.wallet.domain.error`에 둔다. 기존 `InsufficientBalanceException`, `CurrencyMismatchException`과 한자리에 모아, service가 문자열 검사 없이 던지고 이후 ControllerAdvice(014)가 셋을 RFC 7807로 일관되게 매핑한다.

```java
public class AccountNotFoundException extends RuntimeException {
  // TODO: userId를 실어 "어느 계좌가 없는지" 드러내는 생성자
}
```

## 검증

- `mvn -q compile` — 인터페이스/예외가 컴파일되는지.
- 감각 점검(동시성은 008이 실제 PostgreSQL 위에서 고정): 잔고 변경 경로가 `findById`(락 없음)를 쓰면 008의 100건 동시 debit이 단일 분개 쌍으로 수렴하지 못한다 — 반드시 `findByUserIdForUpdate`를 거쳐야 한다. PostgreSQL row lock은 트랜잭션 커밋까지 유지된다.
