# DailyReconciliationJob (ledger drift 탐지)

- 해시: `b1ac969` feat(scheduler): add DailyReconciliationJob for ledger drift detection
- 매일 새벽 전체 ledger 합과 계좌별 스냅샷-원장 일치를 검증하는 배치를 둔다

## 개요

`DailyReconciliationJob`은 per-operation `InvariantChecker`가 개별 쌍만으로는 볼 수 없는 두 가지를 매일 한 번 점검한다. 시스템 전체 차변−대변 = 0과, 각 계좌 스냅샷이 원장 entry 합과 일치하는지다. 후자가 wallet의 "스냅샷 + 증분" 모델(CLAUDE.md "잔고 계산")의 닫는 고리다. 스냅샷이 빠른 read 경로이고 ledger entry가 audit ground truth다.

## 작업 순서

1. `@Scheduled(cron)`으로 매일 03:00 UTC에 돈다.
2. 시스템 전체 net을 검사한다.
3. 계좌마다 두 bucket 스냅샷을 원장 합과 대조한다.
4. Micrometer 카운터/게이지로 상태를 노출한다.

## 작업 내역

### 두 검증

```java
@Scheduled(cron = "${wallet.reconciliation.cron:0 0 3 * * *}")
@Transactional(readOnly = true)
public void reconcile() {
  runs.increment();
  long net = ledgerRepo.netSumAll();
  systemNet.set(net);
  if (net != 0L) {
    violations.increment();
    log.error("RECONCILIATION DRIFT: system-wide ledger DEBIT − CREDIT = {} (expected 0)", net);
  }

  for (Account account : accountRepo.findAll()) {
    verifyBucket(account, BalanceBucket.AVAILABLE, account.available().amount());
    verifyBucket(account, BalanceBucket.LOCKED, account.locked().amount());
  }
}
```

per-operation 리스너가 각 쌍을 국소적으로 보는 반면, 이 배치는 연산을 가로질러 나타나는 drift를 잡는다. 빠진 entry, 손으로 고친 SQL, 오래된 덤프에서의 복원 같은 것이다.

애플리케이션 타임존이 UTC(ADR-0003)라 기본 cron은 03:00 UTC다. 운영자가 `wallet.reconciliation.cron`으로 덮을 수 있다.

### Micrometer 메트릭 셋

- `wallet.reconciliation.runs`: tick당 증가 카운터.
- `wallet.reconciliation.violations`: 발견한 위반당 증가.
- `wallet.reconciliation.net`: 마지막 관측 시스템 net 게이지. 정상 시 0, 0이 아니면 Grafana 이상치.

net 게이지를 받치는 인메모리 `AtomicLong`을 메트릭 레지스트리에 Supplier로 공유한다. Micrometer가 scrape마다 읽어, 마지막 reconciliation이 관측한 값이 다음 실행 전까지 보인다.

`@Transactional(readOnly = true)`라 배치가 순회 중 상태를 실수로 바꾸지 못한다. 순회는 전체 테이블이다. `findAll(Pageable)` 페이지네이션은 wallet 테이블이 충분히 커져 read 트랜잭션이 길게 잡힐 때의 미래 개선으로 남겼다. V1은 가상의 규모를 위한 복잡도를 미루고 단순하게 둔다.

## 결과

매 커밋 후 검사(per-operation)와 매일 배치(전체 합 + 스냅샷 대조)가 정합성을 이중으로 지킨다. 둘 다 위반을 메트릭으로 노출해 운영 알람으로 이어진다.

## 요약

- 매일 03:00 UTC에 전체 net = 0과 계좌별 스냅샷-원장 일치를 검증한다.
- per-operation 리스너가 못 보는 cross-operation drift를 잡는다.
- 카운터 2개 + 게이지 1개로 상태를 Prometheus에 노출한다.
- 전체 테이블 순회이고 페이지네이션은 미래 개선이다.

## 다음 작업

## 핵심 확인

- 스냅샷은 빠른 read, ledger entry는 audit ground truth. 둘의 일치가 이 배치의 핵심.
- `@Transactional(readOnly = true)`로 순회 중 변경을 막는다.
- 전체 테이블 순회는 규모가 커지면 페이지네이션 대상이다(change-cost 참조).

## 기억/설명 Level

### L3 - 백지 설명 대상

- per-operation 검사와 일일 배치의 이중 정합성 구조, 그리고 스냅샷 + 증분 모델에서 배치가 스냅샷-원장 일치를 검증해 닫는 고리

### L2 - 코드/문서 기반 설명 대상

- 시스템 net 검사 + 계좌별 bucket 대조 흐름과 cron/타임존 결정
- Micrometer 게이지를 `AtomicLong` Supplier로 공유해 scrape마다 노출하는 방식

### L1 - 읽고 파악하면 충분한 대상

- 전체 테이블 순회를 V1에서 단순하게 둔 트레이드오프
