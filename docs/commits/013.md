# /internal/v1/wallet REST 표면 노출

- 해시: `e658da1` feat(api): expose /internal/v1/wallet REST surface
- 시스템 내부 서비스가 호출하는 6개 endpoint와 요청/응답 DTO를 노출한다

## 개요

`WalletController`는 시스템의 다른 서비스가 wallet에 들어오는 HTTP 입구다. 모든 경로가 `/internal/v1/wallet` 아래에 있다(ADR-0004). "internal" 세그먼트는 gateway가 외부 클라이언트 트래픽을 여기로 직접 전달하지 않음을 뜻한다. betting-service, settlement-service, admin-api, 그리고 gateway의 내부 read 호출만 온다.

Spring MVC의 `@RestController`는 메서드 반환값을 자동으로 JSON 직렬화한다. C/C++에서 HTTP 핸들러와 직렬화를 손으로 엮는 것과 달리, 어노테이션과 DTO 타입만으로 와이어 형태가 정해진다.

## 작업 순서

1. 6개 endpoint를 `@RestController`에 매핑한다.
2. transaction endpoint는 `Idempotency-Key` 헤더를 받는다.
3. 요청/응답 DTO를 record로 둔다.
4. `@Valid`로 Bean Validation을 건다.

## 작업 내역

### endpoint

| Method | Path | 비고 |
|--------|------|------|
| POST | `/internal/v1/wallet/accounts` | 계좌 생성/반환 |
| GET | `/internal/v1/wallet/accounts/{userId}/balance` | 잔고 조회 |
| POST | `/internal/v1/wallet/transactions/deposit` | Idempotency-Key 필수 |
| POST | `/internal/v1/wallet/transactions/withdraw` | Idempotency-Key 필수 |
| POST | `/internal/v1/wallet/transactions/debit` | Idempotency-Key 필수 |
| POST | `/internal/v1/wallet/transactions/credit` | Idempotency-Key 필수 |

```java
@PostMapping("/transactions/debit")
public WalletOperationResponse debit(@Valid @RequestBody TransactionRequest req,
    @RequestHeader("Idempotency-Key") String idempotencyKey) {
  return WalletOperationResponse.from(
      walletService.debit(new DebitCommand(req.userId(), req.amount(), IdempotencyKey.of(idempotencyKey))));
}
```

transaction endpoint는 `Idempotency-Key`를 평문 String 헤더로 받아 `IdempotencyKey.of(...)`로 변환한다. shared-protocol의 compact 생성자가 와이어 형태(printable ASCII, 128자 이하)를 검증하고, 위반은 `IllegalArgumentException`으로 떠서 다음 커밋의 핸들러가 RFC 7807 400으로 매핑한다.

### DTO

요청/응답을 record로 둔다. `OpenAccountRequest`, `TransactionRequest`(userId, amount), `CreditRequest`(userId, amount, source), `AccountResponse`, `BalanceResponse`, `WalletOperationResponse`. Money/Currency/IdempotencyKey는 shared-protocol에서 오고, Jackson 바인딩이 거기 고정돼 있어 이 API를 소비하는 서비스마다 JSON 형태가 동일하다.

`@Valid @RequestBody`마다 Bean Validation이 돈다. record 컴포넌트의 `@NotNull`이 최상위 필드 누락을 막고, 금액 양수/통화 일치 같은 깊은 검사는 도메인 mutator에 남아 다음 커밋의 핸들러로 떠오른다.

## 결과

컨트롤러는 오류 매핑을 갖지 않는다. Spring의 표준 예외 흐름과 다음 커밋의 ControllerAdvice가 도메인 예외를 ProblemDetail로 일관되게 번역한다.

## 요약

- 6개 endpoint가 `/internal/v1/wallet` 아래 internal API로 노출된다.
- transaction은 `Idempotency-Key` 헤더 필수다.
- DTO는 record이고 shared-protocol 타입으로 서비스 간 JSON 형태를 통일한다.
- 컨트롤러는 오류 매핑 없이 얇게 유지한다.

## 다음 작업

도메인 예외를 RFC 7807 ProblemDetail로 매핑하는 ControllerAdvice를 추가한다.

## 핵심 확인

- 모든 경로는 internal API다(gateway가 외부 트래픽을 직접 전달하지 않음).
- `Idempotency-Key`는 String으로 받아 `IdempotencyKey.of`로 검증한다.
- 컨트롤러에는 오류 매핑이 없다.

## 기억/설명 Level

### L2 - 코드/문서 기반 설명 대상

- endpoint 6개의 책임과 internal API 경로 규약(ADR-0004)
- DTO를 shared-protocol 타입으로 구성해 서비스 간 JSON을 통일하는 방식

### L1 - 읽고 파악하면 충분한 대상

- `@RestController`가 반환값을 자동 JSON 직렬화하는 점
- `Idempotency-Key` 헤더를 String으로 받아 value object로 변환하는 흐름
