# 도메인 단위 테스트 (Account / LedgerEntry / UuidV7)

- 해시: `fe4d81f` test(domain): cover Account arithmetic, ledger pairs, UUID v7 layout
- Spring 컨텍스트 없이 도는 순수 JUnit 테스트로 도메인 산술과 분개 쌍, UUID 비트 레이아웃을 덮는다

## 개요

세 테스트 클래스 모두 Spring 컨텍스트 없이 돈다. 도메인이 결정적(deterministic)이라 수십 ms 안에 끝나고, 이후 통합 테스트가 의존하는 안쪽 루프가 된다. JUnit5 + AssertJ 조합을 쓴다. AssertJ의 `assertThat(x).isEqualTo(y)` 체이닝은 C/C++의 `assert` 매크로보다 실패 메시지가 풍부하다.

## 작업 순서

1. `AccountTest`를 bucket 전이별 4개 nested 그룹으로 나눈다.
2. `LedgerEntryTest`로 `pair()` 팩토리의 invariant를 검증한다.
3. `UuidV7Test`로 RFC 9562 비트 레이아웃과 시간 정렬을 검증한다.

## 작업 내역

### `AccountTest`

`@Nested`로 mutator별 그룹을 나눈다. JUnit5의 nested 클래스는 테스트를 도메인 동작 단위로 묶어 실패 위치를 좁힌다.

```java
@Nested
@DisplayName("moveAvailableToLocked — bet stake staging")
class StakeStaging {

  @Test
  void moves_funds_and_keeps_total_invariant() {
    Account a = funded(10_000);

    a.moveAvailableToLocked(krw(3_000), T1);

    assertThat(a.available()).isEqualTo(krw(7_000));
    assertThat(a.locked()).isEqualTo(krw(3_000));
    assertThat(a.total()).isEqualTo(krw(10_000));
    assertThat(a.updatedAt()).isEqualTo(T1);
  }

  @Test
  void rejects_when_amount_exceeds_available() {
    Account a = funded(2_000);

    assertThatThrownBy(() -> a.moveAvailableToLocked(krw(2_001), T1))
        .isInstanceOf(InsufficientBalanceException.class)
        .extracting("requested", "available")
        .containsExactly(krw(2_001), krw(2_000));
  }
```

네 그룹은 staking(available→locked), locked 반환, 입출금(increase/decrease), 패배 몰수(forfeit)다. 각 그룹이 happy path와 overdraft / 통화 불일치 / 음수 입력 거부를 함께 본다. forfeit는 locked만 줄고 available은 그대로이며 total이 몰수액만큼 감소함을 확인한다.

### `LedgerEntryTest`

`pair()` 팩토리는 모든 service 연산이 거치는 단일 통로라 invariant를 명시적으로 검증한다.

```java
// Tying fields that the unique-constraint pair relies on:
assertThat(pair.debit().idempotencyKey()).isEqualTo(pair.credit().idempotencyKey());
assertThat(pair.debit().operationGroupId()).isEqualTo(pair.credit().operationGroupId());
assertThat(pair.debit().operationGroupId()).isEqualTo(group);

// Entry IDs are independent UUIDs, even though everything else lines up.
assertThat(pair.debit().entryId()).isNotEqualTo(pair.credit().entryId());
```

차변/대변이 키와 그룹 ID를 공유하되 entry_id는 각자 다름을 본다. 비양수 금액은 생성 시점에 거부되고, `TransferLeg`의 compact 생성자가 null 필드를 막는다.

### `UuidV7Test`

```java
assertThat(id.version()).isEqualTo(7);
assertThat(id.variant()).isEqualTo(2); // RFC 9562 variant 10 → java.util.UUID#variant() == 2
```

timestamp를 주입하는 오버로드로 세 개의 오름차순 timestamp가 오름차순 48비트 prefix를 만드는지 본다. B-tree 친화 삽입 패턴이 의존하는 성질이다. 1000번 연속 생성이 모두 distinct함도 확인한다.

## 결과

이 테스트들이 이후 모든 테스트 레이어의 입력이다. 통합 테스트는 `Account`의 도메인 invariant가 성립한다고 가정하고, 부하 테스트는 그 위에 동시성 차원만 검증한다.

## 요약

- 도메인이 결정적이라 Spring 없이 수십 ms에 끝난다.
