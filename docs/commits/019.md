# 부하/증명 테스트 baseline (k6)

- 해시: `3fa464a` test(load): record initial wallet baseline result (dev-host)
- k6 하니스와 첫 dev-host baseline을 박제한다. 처리량 + Level 3 정합성 증명

## 개요

`load-test/` 디렉터리에 k6 하니스와 첫 큐레이션 baseline을 추가한다. k6는 JavaScript로 시나리오를 짜는 부하 테스트 도구다. docker-compose가 PostgreSQL + Redis + Kafka + wallet JAR를 비표준 포트(55432/56379/59092/58081)로 띄워 로컬 dev DB와 충돌하지 않게 한다.

## 작업 순서

1. docker-compose로 격리된 부하 테스트 스택을 띄운다.
2. `seed.js`로 계좌를 만들고 큰 금액을 입금한다.
3. `debit_load.js`로 constant-arrival-rate 처리량을 측정한다.
4. `concurrency.js` / `idempotency.js`로 Level 3 정합성을 증명한다.
5. 결과를 `results/<날짜>/`와 `BEST.md`에 박제한다.

## 작업 내역

### 하니스 구성

```
load-test/
  docker-compose.yml   격리 포트 스택. apache/kafka 3.7 KRaft 모드
  scenarios/
    seed.js            계좌 N개 + 100억 KRW 입금
    debit_load.js      constant-arrival-rate, threshold 내장
    concurrency.js     한 user 100 동시 debit, distinct key
    idempotency.js     같은 Idempotency-Key 100 동시 debit
  results/2026-05-28/  baseline JSON
  results/BEST.md      큐레이션 결과 + 환경 메모
```

`debit_load.js`는 k6 threshold에 목표(`http_req_failed < 0.1%`, `p99 < 50ms`)를 박아 미달 시 exit code가 0이 아니다. CI 게이트로 그대로 쓸 수 있다.

bitnami/kafka가 arm64 manifest를 안 줘서 apache/kafka로 바꿨다. dev host가 Apple Silicon이라는 고정 사실을 이미지 선택 때 의식하지 않아 헛돌았던 지점이다.

### dev-host 측정 (M1 Pro + Docker Desktop 28, 8 CPU)

| Target | Actual | Errors | p50 | p95 | p99/max |
|--------|--------|--------|-----|-----|---------|
| 200 RPS | 200.01 | 0 | 2.84ms | 6.56ms | ≤ 20ms |
| 500 RPS | 500.02 | 0 | 1.75ms | 3.15ms | 19.96ms |
| 1000 RPS | 993.42 | 0 | 4.55ms | 105.30ms | 787.15ms |

500 RPS에서 p95 3.15ms가 이 호스트의 sweet spot이다. 1000 RPS는 Docker CPU 풀이 포화하며 tail이 늘지만 에러는 0이다. 설계(per-user 락 + Redis fast path + outbox)에 구조적 한계는 안 보이고, k6가 노트북을 앞질러 소진한다. production급 하드웨어는 CLAUDE.md의 5000 RPS 목표를 넘겨야 한다.

### Level 3 정합성 증명

- `concurrency.js`: 100개 distinct-key debit(각 100 KRW) → available −10,000, locked +10,000, total 불변. double-spend 0.
- `idempotency.js`: 100개 same-key debit(각 100 KRW) → available −100, locked +100. 정확히 한 번 적용. 나머지 99는 Redis fast path나 `UNIQUE(idempotency_key, side)` 제약에 막혀 원래 operation_group_id를 돌려받는다.

## 결과

처리량과 정합성 두 축의 baseline이 박제됐다. 통합 테스트(008)가 작은 규모로 증명한 것을 부하 도구로 다시 확인했다. production급 측정은 orchestration repo의 e2e 하니스로 미뤘다.

## 요약

- docker-compose 격리 스택 + k6 시나리오 4종.
- dev host에서 500 RPS / p95 3.15ms / 에러 0%.
- Level 3: 100 동시 debit double-spend 0, same-key 100건 단일 적용.
- threshold 내장으로 CI 게이트 가능.
