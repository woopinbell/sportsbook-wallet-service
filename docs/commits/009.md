# Flyway V2 스키마 (outbox_event)

- 해시: `33b74e2` build(flyway): add V2 schema for outbox_event
- 잔고 변경과 같은 커밋 안에서 Kafka 이벤트를 쌓을 transactional outbox 테이블을 만든다

## 개요

transactional outbox는 분산 트랜잭션의 부분 실패를 피하는 패턴이다(ADR-0006). DB 트랜잭션 커밋과 메시지 발행이 따로 일어나면, 커밋은 됐는데 발행이 실패하거나 그 반대가 생긴다. outbox는 이벤트를 잔고 변경과 같은 트랜잭션 안에서 테이블에 쓰고, 별도 publisher가 그 테이블을 읽어 Kafka로 보낸다. 커밋이 원자적이므로 이벤트가 원장 변경과 어긋날 수 없다.

이 커밋은 그 테이블만 만든다. entity와 publisher는 다음 커밋이다. 단계를 쪼개 각 커밋이 한 가지 관심사만 갖게 한다.

## 작업 순서

1. `outbox_event` 테이블에 이벤트 식별 / 발행 상태 컬럼을 둔다.
2. 미발행 행을 빠르게 찾는 partial index를 만든다.

## 작업 내역

### `outbox_event` 테이블

```sql
CREATE TABLE outbox_event (
    event_id      UUID                     PRIMARY KEY,
    topic         VARCHAR(64)              NOT NULL,
    partition_key VARCHAR(64)              NOT NULL,
    schema_name   VARCHAR(64)              NOT NULL,
    payload       BYTEA                    NOT NULL,
    created_at    TIMESTAMP WITH TIME ZONE NOT NULL,
    published_at  TIMESTAMP WITH TIME ZONE
);
```

- `partition_key`: 발행 시 Kafka 레코드의 키. 사용자 UUID 문자열을 담아 한 사용자의 모든 이벤트가 같은 파티션에 가게 한다(ADR-0006).
- `schema_name`: Avro 레코드 이름. publisher가 발행에 필요로 하진 않고 진단용이다.
- `payload BYTEA`: Avro 바이너리 바이트. V1은 Schema Registry 없이 발행하고(ADR-0014) consumer가 shared-protocol의 같은 Avro 클래스를 고정한다.
- `published_at`: ack 받기 전까지 NULL. publisher는 발행된 행을 지우지 않아 downstream replay용으로 남는다.

### partial index

```sql
CREATE INDEX ix_outbox_unpublished ON outbox_event (created_at) WHERE published_at IS NULL;
```

partial index는 조건에 맞는 행만 인덱싱하는 PostgreSQL 기능이다. "미발행 행을 오래된 순으로"가 hot read path인데, 대부분의 행은 곧 발행돼 인덱스에서 빠지므로 인덱스가 작게 유지된다.

## 결과

publisher가 읽을 테이블이 준비됐다. 다음 커밋의 `OutboxEvent` entity가 이 테이블에 매핑된다.

## 요약

- outbox는 잔고 변경과 같은 트랜잭션에 이벤트를 써서 발행 원자성을 보장한다.
- payload는 Avro 바이너리이고 Schema Registry는 없다(ADR-0014).
- partial index로 미발행 행 스캔을 작게 유지한다.
- 발행된 행은 지우지 않아 replay 가능하다.

## 다음 작업

## 핵심 확인

- 이벤트는 잔고 변경과 같은 트랜잭션에 들어가야 원자성이 성립한다.
- `published_at IS NULL` partial index가 미발행 스캔의 hot path다.

## 기억/설명 Level

### L3 - 백지 설명 대상

- transactional outbox가 DB 커밋과 메시지 발행의 원자성을 어떻게 보장하는지(같은 트랜잭션에 이벤트를 쓰고 별도 publisher가 흘림)

### L2 - 코드/문서 기반 설명 대상

- partition_key에 userId를 담아 사용자별 파티션 순서를 보존하는 설계
- partial index가 미발행 행만 인덱싱해 작게 유지되는 이유

### L1 - 읽고 파악하면 충분한 대상

- 발행된 행을 지우지 않고 남겨 replay에 쓰는 운영 결정
