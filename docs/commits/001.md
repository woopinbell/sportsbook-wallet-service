# Account 집합체 도입 (available + locked)

- 해시: `34c59e6` feat(domain): introduce Account aggregate with available + locked balances
- 사용자당 한 계좌를 사용 가능 / 잠금 두 bucket으로 나눠 들고, 의도 이름이 붙은 mutator로만 잔고를 바꾼다

## 개요

`Account`는 시스템에서 유일하게 비관적 락(pessimistic lock) 아래에서 잔고가 바뀌는 entity다. 잔고를 `available`(쓸 수 있는 돈)과 `locked`(베팅 노출에 묶인 돈) 두 bucket으로 나눈 것이 이 repo의 핵심 모델 결정이다. 베팅이 접수되면 stake를 계좌 밖으로 빼지 않고 available에서 locked로 옮긴다. 돈이 계좌 안에 남아 있어야 운영 액션(노출 강제 환불)과 일일 reconciliation에서 추적이 된다.

한 계좌는 한 통화만 든다(one account = one currency). KRW와 USD를 둘 다 지원하지만 계좌 단위로는 단일 통화다.

## 작업 순서

1. `Money`(shared-protocol record)를 JPA가 받을 수 있는 `EmbeddedMoney` wrapper로 감싼다.
2. `Account` entity에 available / locked 두 `@Embedded` bucket과 `@Version`을 둔다.
3. bucket 전이를 의도 이름 mutator로 노출한다.
4. 각 mutator가 금액 양수 / 통화 일치 / 잔고 충분을 검증하게 한다.
5. 도메인 예외(`InsufficientBalanceException`, `CurrencyMismatchException`)를 둔다.

## 작업 내역

### `EmbeddedMoney.java`

Hibernate 6은 다른 모듈의 record(shared-protocol `Money`)를 `@Embeddable`로 직접 바인딩하지 못한다. 영속성 경계 전용 wrapper를 두되, 도메인 API는 여전히 `Money`를 주고받는다. C/C++이라면 struct를 그대로 재사용했을 자리에, ORM의 reflection 매핑 제약 때문에 어댑터 한 겹이 들어간다.

```java
@Embeddable
public class EmbeddedMoney {

  @Column(nullable = false)
  private long amount;

  @Enumerated(EnumType.STRING)
  @Column(nullable = false, length = 3)
  private Currency currency;

  protected EmbeddedMoney() {
    // Required by JPA.
  }

  public EmbeddedMoney(long amount, Currency currency) {
    this.currency = Objects.requireNonNull(currency, "currency");
    this.amount = amount;
  }

  public static EmbeddedMoney of(Money money) {
    Objects.requireNonNull(money, "money");
    return new EmbeddedMoney(money.amount(), money.currency());
  }

  public Money toMoney() {
    return new Money(amount, currency);
  }
```

### `Account.java`

mutator는 산술 이름(`add`)이 아니라 도메인 의도 이름을 쓴다. 호출부가 실제 자금 흐름으로 읽힌다.

```java
public void moveAvailableToLocked(Money amount, Instant now) {
  requirePositive(amount);
  requireSameCurrency(amount);
  if (available.amount() < amount.amount()) {
    throw new InsufficientBalanceException(userId, amount, available.toMoney());
  }
  available =
      new EmbeddedMoney(Math.subtractExact(available.amount(), amount.amount()), currency());
  locked = new EmbeddedMoney(Math.addExact(locked.amount(), amount.amount()), currency());
  updatedAt = now;
}
```

mutator 다섯 개가 자금 흐름에 1:1로 대응한다. `moveAvailableToLocked`(베팅 stake), `moveLockedToAvailable`(void/승리 stake 반환), `increaseAvailable`(입금/수익 payout), `decreaseAvailable`(출금), `forfeitLocked`(패배 stake 몰수). 모든 산술은 `Math.addExact`/`subtractExact`로 overflow 시 예외를 던진다. shared-protocol `Money`의 overflow 정책과 같다.

`@Version` 필드는 잔고가 아닌 메타데이터 경합용이다. 잔고 경합은 repository 레이어의 `SELECT FOR UPDATE`가 해결한다.

## 결과

이후 service 레이어는 bucket 내부 메모리 배치를 신경 쓰지 않고 의도 이름 mutator 위에서 자금 흐름을 조립할 수 있다. 모든 mutator가 진입 시점에 양수/통화/잔고를 검증하므로, 음수 잔고나 통화 혼합이 entity 안으로 들어올 수 없다.

## 요약

- available + locked 2-bucket으로 베팅 노출을 계좌 안에서 분리한다.
- `EmbeddedMoney`는 JPA 매핑 전용 wrapper이고 도메인 API는 `Money`를 쓴다.
- mutator는 양수/통화/잔고 세 invariant를 상태 변경 전에 검증한다.

## 핵심 확인

- 한 계좌는 한 통화만 든다. 통화 불일치는 `CurrencyMismatchException`.
- 잔고 산술은 `Math.addExact`/`subtractExact`로 overflow를 예외화한다.
- `@Version`은 메타데이터 경합용이고, 잔고 경합은 비관적 락이 막는다.

## 기억/설명 Level

### L3 - 백지 설명 대상

- available / locked 2-bucket으로 베팅 노출을 계좌 밖으로 옮기지 않고 분리하는 모델 결정과, 그 결정이 운영 액션과 reconciliation 추적을 가능하게 하는 이유

### L2 - 코드/문서 기반 설명 대상

- 의도 이름 mutator 다섯 개와 자금 흐름의 1:1 대응
- `EmbeddedMoney` wrapper가 필요한 이유(Hibernate가 외부 모듈 record를 `@Embeddable`로 못 받음)
- 양수/통화/잔고 세 invariant를 상태 변경 전에 검증하는 순서
