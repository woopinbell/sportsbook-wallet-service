# Flyway V1 스키마 (account + ledger_entry)

- 해시: `ec94e6d` build(flyway): add V1 schema for account + ledger_entry
- 앞선 두 도메인 커밋의 JPA 매핑을 PostgreSQL 테이블로 물질화하고 제약과 인덱스를 SQL로 박는다

## 개요

Flyway는 SQL 파일 기반 마이그레이션 도구다. `V<버전>__<설명>.sql` 파일을 버전 순으로 적용하고, 애플리케이션 시작 시 자동으로 미적용 마이그레이션을 검증한다. C/C++ 프로젝트에는 보통 없는 계층이다. 스키마 변경 이력 자체가 코드 저장소에 남고 PR 리뷰 대상이 된다.

이 커밋은 `account`와 `ledger_entry` 두 테이블을 만든다. 두 테이블 사이에 외래 키(FK)는 두지 않는다.

## 작업 순서

1. `account` 테이블에 두 bucket 컬럼과 통화/음수 방지 CHECK 제약을 둔다.
2. `ledger_entry` 테이블에 멱등성 복합 unique 제약 두 개를 둔다.
3. 읽기 경로용 인덱스 두 개를 만든다.
4. 모든 TIMESTAMP를 WITH TIME ZONE으로 둔다(ADR-0003).

## 작업 내역

### `account` 테이블

CHECK 제약 세 개가 도메인 invariant를 DB에 한 번 더 박는다. Hibernate 매핑이 잘못되거나 누군가 직접 UPDATE를 해도 테이블이 도메인이 거부할 상태로 가지 못하게 한다.

```sql
CONSTRAINT account_currency_match   CHECK (available_currency = locked_currency),
CONSTRAINT account_available_nonneg CHECK (available_amount >= 0),
CONSTRAINT account_locked_nonneg    CHECK (locked_amount    >= 0)
```

### `ledger_entry` 테이블

복합 unique 제약 두 개가 matched-pair 멱등성 패턴을 인코딩한다.

```sql
CONSTRAINT uk_ledger_entry_idem_side  UNIQUE (idempotency_key, side),
CONSTRAINT uk_ledger_entry_group_side UNIQUE (operation_group_id, side)
```

`account_id`에서 `account.user_id`로 가는 FK는 두지 않는다. ledger는 시스템 계좌(HOUSE, EXTERNAL_PAYMENT)도 상대로 쓰는데 이들은 account 테이블에 행이 없기 때문이다. account_id가 실제 사용자 계좌이거나 고정 시스템 ID 중 하나라는 보장은 service 레이어가 한다.

bucket / side / reason 컬럼에는 CHECK로 enum 값 집합을 박았다. 직접 INSERT가 알 수 없는 값을 넣지 못하고 `psql \d` 출력에 스키마가 자기 문서화된다.

### 인덱스

```sql
CREATE INDEX ix_ledger_entry_account_created ON ledger_entry (account_id, created_at);
CREATE INDEX ix_ledger_entry_idem_key        ON ledger_entry (idempotency_key);
```

`(account_id, created_at)`는 reconciliation과 invariant listener가 쓰는 "이 계좌의 모든 분개를 시간 순으로" 스캔용이다. `(idempotency_key)`는 Redis 미스 시 fast path와 재시도 응답 조회용이다.

## 결과

`spring.jpa.hibernate.ddl-auto=validate` 설정과 맞물려, 애플리케이션 시작 시 Flyway가 스키마를 만들고 Hibernate가 entity 매핑이 실제 테이블과 맞는지 검증한다. 매핑과 SQL이 어긋나면 부팅이 실패하므로 둘의 불일치가 런타임 전에 잡힌다.

## 요약

- CHECK 제약으로 도메인 invariant(통화 일치, 음수 금지)를 DB에 이중으로 박았다.
- matched-pair 멱등성은 `(idempotency_key, side)` 복합 unique로 SQL에 인코딩된다.
- 시스템 계좌 때문에 ledger_entry에서 account로 가는 FK는 두지 않는다.
- 모든 시각 컬럼은 WITH TIME ZONE이다.

## 핵심 확인

- `ddl-auto=validate`라 Flyway가 만든 스키마와 entity 매핑이 부팅 시 대조된다.
- ledger_entry는 FK가 없다. account_id 유효성은 service가 보장한다.
- WITH TIME ZONE이 아니면 JDBC 경계에서 offset이 잘린다(ADR-0003).

## 기억/설명 Level

### L2 - 코드/문서 기반 설명 대상

- CHECK 제약으로 도메인 invariant를 DB에 이중으로 박는 defence-in-depth 패턴
- ledger_entry에 FK를 두지 않는 이유(시스템 계좌가 account 행을 갖지 않음)
- 두 인덱스가 각각 어느 읽기 경로(reconciliation 스캔 / 멱등성 조회)를 받치는지

### L1 - 읽고 파악하면 충분한 대상

- Flyway의 `V<버전>__<설명>.sql` 명명과 시작 시 자동 적용/검증 동작
- bucket/side/reason CHECK가 enum 값 집합을 SQL에 박아 자기 문서화하는 점
