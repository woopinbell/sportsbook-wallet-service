# OutboxEvent entity + 스케줄 publisher

- 해시: `2d788d2` feat(outbox): introduce OutboxEvent entity + scheduled publisher
- outbox 행 entity와 1초 주기로 미발행 행을 Kafka로 흘리는 스케줄 publisher, 그리고 Kafka producer 설정을 만든다

## 개요

앞 커밋이 만든 outbox 테이블에 매핑되는 `OutboxEvent` entity와, 그 테이블을 주기적으로 비우는 `OutboxPublisher`를 도입한다. 잔고를 바꾸는 모든 연산은 `OutboxEvent.pending()`으로 행을 만들어 분개 쌍과 같은 트랜잭션에 저장한다.

## 작업 순서

1. `OutboxEvent` entity를 write-once에 가깝게 만든다(`publishedAt`만 변경 가능).
2. `OutboxEventRepository.findUnpublished(Pageable)`로 배치 조회를 노출한다.
3. `OutboxPublisher`를 `@Scheduled` 1초 주기로 돌린다.
4. `KafkaConfig`로 `(String, byte[])` producer를 wire한다.

## 작업 내역

### `OutboxEvent`

대부분 필드가 write-once다. `publishedAt`만 변경 가능하고 `markPublished()`는 멱등이라 중복 발행 응답이 행을 흔들지 못한다. `payload`는 생성과 읽기 양쪽에서 방어적으로 clone해 BYTEA 버퍼를 Java 측에서 불변으로 유지한다.

```java
/** Marks the row as successfully published. Idempotent — second call is a no-op. */
public void markPublished(Instant at) {
  if (publishedAt == null) {
    publishedAt = Objects.requireNonNull(at, "at");
  }
}
```

### `OutboxPublisher`

```java
@Scheduled(fixedDelayString = "${wallet.outbox.poll-interval-ms:1000}")
@Transactional
public void publishPending() {
  List<OutboxEvent> batch = repository.findUnpublished(PageRequest.of(0, BATCH_SIZE));
  if (batch.isEmpty()) {
    return;
  }
  for (OutboxEvent event : batch) {
    if (sendBlocking(event)) {
      event.markPublished(clock.instant());
    }
  }
}
```

매 tick마다 미발행 행을 오래된 순으로 최대 `BATCH_SIZE`(기본 100)개 가져오고, 각각 Kafka로 보내 ack를 기다린(5초 timeout) 뒤 `published_at`을 찍는다. dirty checking으로 entity 변경이 메서드 종료 시 커밋된다. 실패하면 로그만 남기고 행을 미발행으로 둬서 다음 tick이 재시도한다. 이 retry-until-acked가 outbox를 at-least-once로 만든다.

`Pageable`로 배치를 제한한 것은 backlog가 스케줄러를 멈추지 못하게 하기 위함이다.

### `KafkaConfig`

```java
props.put(ProducerConfig.ACKS_CONFIG, "all");
props.put(ProducerConfig.ENABLE_IDEMPOTENCE_CONFIG, true);
props.put(ProducerConfig.MAX_IN_FLIGHT_REQUESTS_PER_CONNECTION, MAX_IN_FLIGHT_REQUESTS);
```

value를 `byte[]`로 둔 것은 outbox 행이 이미 Avro 인코딩된 payload를 들고 있기 때문이다. `acks=all` + `enable.idempotence=true` + `max.in.flight=5`(idempotent producer의 Confluent 권장 상한) 조합으로 브로커 내부 재시도가 레코드를 중복시키지 않고, outbox의 retry-until-acked와 합쳐져 파티션 순서를 보존하는 at-least-once가 된다.

## 결과

미발행 outbox 행이 1초 주기로 Kafka로 흘러간다. `WalletService`는 아직 outbox에 쓰지 않는다. 그 배선은 다음 커밋으로 미뤄 각 단계가 한 관심사만 갖게 한다.

## 요약

- `OutboxEvent`는 `publishedAt`만 변경 가능한 write-once entity다.
- publisher는 1초 주기 배치로 미발행 행을 Kafka에 보내고 ack 후 `published_at`을 찍는다.
- 실패 행은 다음 tick이 재시도한다(at-least-once).
- producer는 idempotence + acks=all로 중복 없는 파티션 순서를 보존한다.

## 다음 작업

`WalletService`가 각 연산에서 outbox에 이벤트를 쓰도록 배선한다.

## 핵심 확인

- publisher 실패는 행을 미발행으로 두고 다음 tick이 재시도한다.
- 발행은 at-least-once다. 중복은 consumer의 멱등성 모델이 흡수한다.
- 테스트에서는 `wallet.outbox.poll-interval-ms`를 크게 잡아 자동 tick을 멈춘다.

## 기억/설명 Level

### L3 - 백지 설명 대상

- outbox publisher의 retry-until-acked 루프가 at-least-once 전달을 만드는 메커니즘과, idempotent producer가 중복/순서를 어떻게 막는지

### L2 - 코드/문서 기반 설명 대상

- `@Scheduled` + `Pageable` 배치로 backlog가 스케줄러를 멈추지 않게 하는 구성
- `markPublished` 멱등성과 payload 방어적 clone

### L1 - 읽고 파악하면 충분한 대상

- value를 `byte[]`로 두는 이유(행이 이미 Avro 바이트를 들고 있음)
