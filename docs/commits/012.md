# outbox 통합 테스트 (원자성 + end-to-end 발행)

- 해시: `6d5c05c` test(outbox): cover atomic ledger + outbox write and end-to-end publish
- 분개와 outbox가 같은 트랜잭션에 쓰이는지, publisher가 embedded Kafka로 흘려보내는지를 검증한다

## 개요

`OutboxIntegrationTest`가 transactional outbox를 PostgreSQL + Redis(Testcontainers) + embedded Kafka 브로커(spring-kafka-test) 위에서 검증한다. 다섯 테스트로 발행 형태별 하나씩과 end-to-end drain을 본다.

## 작업 순서

1. `@EmbeddedKafka`로 인메모리 브로커를 띄운다.
2. 자동 발행 스케줄을 24시간으로 밀어 테스트가 발행 시점을 직접 통제한다.
3. 각 연산 후 outbox 행을 읽어 Avro payload를 디코딩해 검증한다.
4. `publishPending()`을 직접 호출해 brokered 발행을 본다.

## 작업 내역

### embedded Kafka 배선

PostgreSQL + Redis 컨테이너와 `@DynamicPropertySource` 주입은 008과 같다. 추가된 것은 `@EmbeddedKafka`로 세 토픽을 미리 만들고 broker 주소를 `spring.kafka.bootstrap-servers`에 주입하는 부분이다.

```java
@SpringBootTest
@Testcontainers
@EmbeddedKafka(
    partitions = 1,
    topics = {
      WalletEventFactory.TOPIC_DEBITED,
      WalletEventFactory.TOPIC_CREDITED,
      WalletEventFactory.TOPIC_DEBIT_FAILED
    },
    bootstrapServersProperty = "spring.kafka.bootstrap-servers")
@ActiveProfiles("test")
class OutboxIntegrationTest {
  @Container static PostgreSQLContainer<?> postgres = new PostgreSQLContainer<>("postgres:16-alpine");
  @Container static GenericContainer<?> redis = new GenericContainer<>("redis:7-alpine").withExposedPorts(6379);
  // @DynamicPropertySource: datasource url/username/password + redis host/port (008과 동일)
}
```

토픽 상수를 `WalletEventFactory`에서 가져와 producer와 test가 같은 이름을 공유한다. `bootstrapServersProperty`가 브로커 주소를 프로퍼티로 흘려 `KafkaConfig`의 producer가 embedded broker에 붙는다.

### 자동 tick 멈추기

`OutboxPublisher`의 `@Scheduled` drain을 `application-test.yml`에서 24시간으로 park한다.

```yaml
wallet:
  outbox:
    # @Scheduled drain을 테스트 중 끈다. 각 테스트가 publishPending()을 직접 불러 발행 시점을 통제.
    poll-interval-ms: 86400000
```

park하지 않으면 write와 assertion 사이에 tick이 끼어들어 `published_at`을 미리 찍어버리고, "쓴 직후엔 미발행"이라는 1·4번 검증이 거짓 음성이 된다.

### 다섯 테스트

1번은 outbox 행이 분개 쌍과 같은 트랜잭션에 남는지를, payload를 디코딩해 `ledger_tx_id == operation_group_id`까지 본다.

```java
@Test
void debit_success_lands_walletDebited_payload() {
  DebitCommand cmd = new DebitCommand(userId, krw(5_000), key("bet"));
  WalletOperationResult result = walletService.debit(cmd);

  OutboxEvent row = findUnpublishedFor(cmd.idempotencyKey().value(), WalletEventFactory.SCHEMA_DEBITED);
  assertThat(row.partitionKey()).isEqualTo(userId.toString());
  assertThat(row.publishedAt()).isNull();

  WalletDebited decoded = AvroSerializer.deserialize(row.payload(), WalletDebited.class);
  assertThat(decoded.getAmount().getAmount()).isEqualTo(5_000L);
  assertThat(decoded.getIdempotencyKey()).isEqualTo(cmd.idempotencyKey().value());
  assertThat(decoded.getLedgerTxId()).isEqualTo(result.operationGroupId().toString());
}
```

2번 HOUSE_POOL credit → `WalletCredited` reason=PAYOUT. 3번 USER_LOCKED credit → reason=REFUND. 4번은 메인 트랜잭션이 롤백돼도 `publishDebitFailure`의 별도 트랜잭션 덕에 실패 행이 살아남는지를 본다.

```java
@Test
void debit_failure_lands_walletDebitFailed_after_rollback() {
  DebitCommand cmd = new DebitCommand(userId, krw(200_000), key("overdraft"));
  assertThatThrownBy(() -> walletService.debit(cmd)).isInstanceOf(InsufficientBalanceException.class);

  OutboxEvent row = findUnpublishedFor(cmd.idempotencyKey().value(), WalletEventFactory.SCHEMA_DEBIT_FAILED);
  WalletDebitFailed decoded = AvroSerializer.deserialize(row.payload(), WalletDebitFailed.class);
  assertThat(decoded.getReason()).isEqualTo(WalletDebitFailureReason.INSUFFICIENT_BALANCE);
}
```

5번은 `publishPending()`이 미발행 행을 embedded 브로커로 흘리고 `published_at`이 non-null이 되며, `wallet.debited.v1`을 구독한 consumer가 key=userId로 레코드를 받아 producer가 쓴 Avro 형태로 역직렬화되는지를 본다.

```java
@Test
void publisher_drains_pending_and_kafka_receives_the_record() {
  walletService.debit(cmd);
  outboxPublisher.publishPending();

  assertThat(findFor(key, WalletEventFactory.SCHEMA_DEBITED).publishedAt()).isNotNull();
  try (Consumer<String, byte[]> consumer = newConsumer()) {
    consumer.subscribe(List.of(WalletEventFactory.TOPIC_DEBITED));
    ConsumerRecords<String, byte[]> polled = consumer.poll(Duration.ofSeconds(10));
    ConsumerRecord<String, byte[]> match = findRecordForKey(polled, userId.toString()).orElseThrow();
    assertThat(AvroSerializer.deserialize(match.value(), WalletDebited.class).getIdempotencyKey())
        .isEqualTo(cmd.idempotencyKey().value());
  }
}
```

행 조회는 `outboxRepo.findAll()`을 payload의 `idempotencyKey` + `schemaName`으로 거르는 `findUnpublishedFor` / `findFor` helper로 한다.

### `AvroSerializer.deserialize`

기존 `serialize`의 짝으로 작은 `deserialize`를 추가해 테스트가 매번 `DatumReader`를 새로 만들지 않고 payload 내용을 검증한다.

```java
public static <T extends SpecificRecord> T deserialize(byte[] payload, Class<T> type) {
  try {
    DatumReader<T> reader = new SpecificDatumReader<>(type);
    return reader.read(null, DecoderFactory.get().binaryDecoder(payload, null));
  } catch (IOException e) {
    throw new IllegalStateException("Failed to deserialize Avro record " + type.getName(), e);
  }
}
```

역직렬화 측은 현재 테스트만 쓴다. 프로덕션 이 서비스는 발행만 하지만, codec 쌍을 한자리에 둬서 같은 payload를 소비할 downstream 서비스가 발견하기 쉽게 한다.

## 결과

스위트 전체 31개가 깨끗이 통과했다. 종료 시점 Kafka warning은 embedded 브로커가 JUnit 생명주기 끝에서 임시 로그 디렉터리를 정리하며 내는 것으로, 테스트 실패가 아니다.

## 요약

- 자동 발행 tick을 park해 테스트가 발행 시점을 통제한다.
- 다섯 테스트가 발행 형태별 payload와 end-to-end drain을 본다.
- 실패 이벤트가 메인 롤백에도 남음을 확인한다.
- `deserialize` 짝을 추가해 payload 검증과 downstream 발견성을 돕는다.
