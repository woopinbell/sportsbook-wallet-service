# outbox 통합 테스트 (원자성 + end-to-end 발행)

- 해시: `ed9cbe4` test(outbox): cover atomic ledger + outbox write and end-to-end publish
- 분개와 outbox가 같은 트랜잭션에 쓰이는지, publisher가 embedded Kafka로 흘려보내는지를 검증한다

## 개요

`OutboxIntegrationTest`가 transactional outbox를 PostgreSQL + Redis(Testcontainers) + embedded Kafka 브로커(spring-kafka-test) 위에서 검증한다. 다섯 테스트로 발행 형태별 하나씩과 end-to-end drain을 본다.

## 작업 순서

1. `@EmbeddedKafka`로 인메모리 브로커를 띄운다.
2. 자동 발행 스케줄을 24시간으로 밀어 테스트가 발행 시점을 직접 통제한다.
3. 각 연산 후 outbox 행을 읽어 Avro payload를 디코딩해 검증한다.
4. `publishPending()`을 직접 호출해 brokered 발행을 본다.

## 작업 내역

### 자동 tick 멈추기

`OutboxPublisher`의 `@Scheduled`를 `application-test.yml`에서 `wallet.outbox.poll-interval-ms`로 24시간에 park한다. 안 그러면 write와 assertion 사이에 tick이 끼어들어 `published_at`을 미리 찍어버려 "쓴 직후엔 미발행"이라는 검증이 거짓 음성이 된다.

### 다섯 테스트

1. debit 성공 → `WalletDebited` 행이 분개 쌍과 같은 트랜잭션에 남는다. partition_key가 userId 문자열, `published_at`이 NULL, payload가 같은 amount/currency/idempotency_key로 디코딩되고, ledger_tx_id가 service가 돌려준 operation_group_id와 일치한다.
2. HOUSE_POOL credit → `WalletCredited` reason=PAYOUT.
3. USER_LOCKED credit → `WalletCredited` reason=REFUND.
4. InsufficientBalance → 메인 트랜잭션이 롤백돼도 `WalletDebitFailed` 행이 reason=INSUFFICIENT_BALANCE로 존재한다(`publishDebitFailure`의 별도 트랜잭션).
5. `publishPending()`이 미발행 행을 embedded 브로커로 흘리고 `published_at`이 non-null이 되며, `wallet.debited.v1`을 구독한 consumer가 key=userId로 레코드를 받는다. 받은 바이트가 producer가 쓴 Avro 형태로 역직렬화된다.

### `AvroSerializer.deserialize`

기존 `serialize`의 짝으로 작은 `deserialize`를 추가해 테스트가 매번 `DatumReader`를 새로 만들지 않고 payload 내용을 검증한다. 역직렬화 측은 현재 테스트만 쓴다. 프로덕션 이 서비스는 발행만 하지만, codec 쌍을 한자리에 둬서 같은 payload를 소비할 downstream 서비스가 발견하기 쉽게 한다.

## 결과

스위트 전체 31개가 깨끗이 통과했다. 종료 시점 Kafka warning은 embedded 브로커가 JUnit 생명주기 끝에서 임시 로그 디렉터리를 정리하며 내는 것으로, 테스트 실패가 아니다.

## 요약

- 자동 발행 tick을 park해 테스트가 발행 시점을 통제한다.
- 다섯 테스트가 발행 형태별 payload와 end-to-end drain을 본다.
- 실패 이벤트가 메인 롤백에도 남음을 확인한다.
- `deserialize` 짝을 추가해 payload 검증과 downstream 발견성을 돕는다.
