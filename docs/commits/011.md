# 결과 이벤트 발행 배선 (outbox 경유)

- 해시: `e35d29f` feat(events): publish wallet outcome events through the transactional outbox
- 각 연산이 분개 쌍과 같은 트랜잭션에서 Avro 이벤트를 outbox에 쓰도록 배선하고, 실패 이벤트는 별도 트랜잭션에서 남긴다

## 개요

`WalletService`가 이제 saga에 참여하는 각 연산마다 분개 쌍과 같은 DB 트랜잭션에서 `OutboxEvent`를 쓴다. 원자적 커밋이 핵심이다. publisher가 이후 어떻게 동작하든 이벤트가 원장 변경과 어긋날 수 없다.

## 작업 순서

1. `WalletEventFactory`로 Avro 이벤트 생성과 토픽/키 규약을 분리한다.
2. debit/credit 성공 시 같은 트랜잭션에서 outbox에 쓴다.
3. debit 실패 시 별도 트랜잭션에서 실패 이벤트를 남긴다.
4. `AvroSerializer`로 바이너리 인코딩을 캡슐화한다.

## 작업 내역

### 세 이벤트 형태

shared-protocol이 생성한 Avro 클래스를 쓴다.

- `WalletDebited`: 베팅 stake debit 성공. userId, amount, idempotencyKey, ledgerTxId(쌍의 operation_group_id), occurredAt.
- `WalletCredited`: credit 성공. reason enum이 `CreditCommand.Source`를 매핑한다. USER_LOCKED→REFUND, HOUSE_POOL→PAYOUT.
- `WalletDebitFailed`: debit 거부. 예외를 매핑한다. InsufficientBalance→INSUFFICIENT_BALANCE, AccountNotFound→ACCOUNT_NOT_FOUND, CurrencyMismatch→CURRENCY_MISMATCH.

deposit과 withdraw는 V1에서 이벤트를 발행하지 않는다. 베팅 saga가 보지 않는 운영성 연산이고, `WalletCredited`의 reason enum(PAYOUT/VOID/REFUND)에 맞는 값이 없다. 필요해지면 새 이벤트 타입을 추가한다.

### 성공 경로와 실패 경로의 트랜잭션 차이

성공 이벤트는 분개 쌍을 쓰는 `TransactionTemplate` 블록 안에서 쓰여 원장 커밋을 공유한다. 실패 이벤트는 별도 fresh 트랜잭션(`publishDebitFailure`)에서 쓴다. 원래 트랜잭션이 롤백되므로 거기서 쓰면 이벤트도 같이 사라지기 때문이다. 트리거한 예외는 caller가 다시 던진다.

```java
} catch (InsufficientBalanceException e) {
  publishDebitFailure(cmd, WalletDebitFailureReason.INSUFFICIENT_BALANCE);
  throw e;
}
```

별도 트랜잭션이 허용되는 이유는 실패가 hot path가 아니고(전체 debit 중 소수여야 함), 대안인 "롤백 밖으로 side-channel payload를 빼는" 방식이 entity manager를 복잡하게 만들기 때문이다.

### `WalletEventFactory`와 토픽 규약

```
topic                    schema             partition_key
wallet.debited.v1        WalletDebited      userId
wallet.credited.v1       WalletCredited     userId
wallet.debit-failed.v1   WalletDebitFailed  userId
```

모든 wallet 이벤트가 userId를 파티션 키로 공유해 토픽에서 사용자별 순서가 보존된다(ADR-0006). `AvroSerializer`는 작은 바이너리 인코딩 helper다. V1은 Schema Registry 없이 발행하고, 스키마 id는 producer/consumer 양쪽의 shared-protocol 생성 `SpecificRecord`에 암묵적으로 들어 있다(ADR-0014).

### Avro logical type 마찰

`occurredAt`은 `timestamp-millis` logical type인데 shared-protocol이 jsr310으로 생성돼 setter가 `long`이 아니라 `Instant`를 받는다. `now.toEpochMilli()`를 넘겼다가 컴파일 에러가 났고 `now`를 그대로 넘기는 것으로 고쳤다.

## 결과

기존 통합 테스트 8개가 여전히 통과한다. Kafka 슬라이스를 제외한 테스트에서 producer가 없는 브로커로 재시도하는 종료 로그 noise가 보이지만, publisher가 로그만 남기고 넘어가므로 테스트 실패로는 올라오지 않는다.

## 요약

- 세 이벤트(Debited/Credited/DebitFailed)를 분개와 같은 트랜잭션에서 outbox에 쓴다.
- 실패 이벤트만 별도 fresh 트랜잭션에서 쓴다(원래 트랜잭션이 롤백되므로).
- deposit/withdraw는 V1에서 발행하지 않는다(맞는 reason enum 없음).
- 모든 이벤트는 userId 파티션 키로 사용자별 순서를 보존한다.

## 다음 작업

outbox 원자성과 end-to-end 발행을 embedded Kafka로 테스트한다.

## 핵심 확인

- 성공 이벤트는 원장과 같은 트랜잭션, 실패 이벤트는 별도 트랜잭션.
- deposit/withdraw 이벤트 미발행은 의도된 V1 한계다.
- `occurredAt` setter는 `Instant`를 받는다(jsr310 생성).

## 기억/설명 Level

### L3 - 백지 설명 대상

- 성공 이벤트를 원장과 같은 트랜잭션에, 실패 이벤트를 별도 트랜잭션에 쓰는 이유와 그 트레이드오프
- outbox에 이벤트를 원자적으로 쓰는 것이 saga 부분 실패를 어떻게 막는지

### L2 - 코드/문서 기반 설명 대상

- `WalletEventFactory`의 토픽/스키마/파티션 키 규약과 `CreditCommand.Source`→reason 매핑
- deposit/withdraw 미발행 결정의 근거(shared-protocol enum 한계)

### L1 - 읽고 파악하면 충분한 대상

- `AvroSerializer`가 Schema Registry 없이 바이너리 인코딩을 캡슐화하는 점
- `occurredAt`의 jsr310 logical type 매핑 마찰
