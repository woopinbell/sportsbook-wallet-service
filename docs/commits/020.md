# 패배 스테이크 forfeit (locked → house)

- 해시: `eddb844` feat(wallet): forfeit a lost bet's stake from locked to the house
- 베팅이 졌을 때 user의 locked 스테이크를 HOUSE로 capture하는 멱등 연산을 추가해 LOST 정산의 빠진 wallet leg를 채운다 (Phase 5 통합)

## 개요

이 커밋은 dev 단계가 아니라 **Phase 5 전체 스택 통합(orchestration e2e)에서 드러난 결함**의 wallet 몫이다.

settlement의 정산 결과는 WON / LOST / PUSH / VOID 네 가지인데, wallet에는 그중 **LOST에 대응하는 연산이 없었다**. 베팅 접수 시 스테이크는 `BET_DEBIT`로 user의 `available`에서 `locked`로 옮겨져 묶인다(staging). 이긴 베팅은 그 스테이크를 `BET_REFUND`로 locked→available 되돌리고 수익을 `BET_PAYOUT`으로 house→user 지급한다. push/void도 `BET_REFUND`로 전액 환불한다. 그런데 **진 베팅은** 묶인 스테이크를 회수하는 leg가 없어, 사용자가 잃은 돈이 영영 user의 locked bucket에 묶여 있었다 — house는 수금하지 못하고 user 잔고도 정리되지 않았다.

도메인에는 이미 `Account.forfeitLocked` mutator와 `HOUSE` 시스템 계좌가 있었지만 **어떤 연산에도 배선돼 있지 않았다**. 설계 시점에 자리는 만들어 뒀는데 LOST 경로가 그 자리를 호출하지 않은 것이다. 이 커밋이 그 자리를 채운다.

## 작업 순서

1. `BET_FORFEIT` ledger reason을 추가한다(`LedgerReason` enum).
2. `ledger_entry_reason_valid` CHECK 제약을 V3 마이그레이션으로 확장한다.
3. `WalletService.forfeit`를 기존 멱등 primitive(`runIdempotent` + `writePair`) 위에 올린다.
4. `ForfeitCommand`와 `POST /internal/v1/wallet/transactions/forfeit` endpoint를 추가한다.
5. 통합 테스트로 locked→house capture, over-locked 거절, 합 = 0 invariant를 고정한다.

## 작업 내역

### 정산 매트릭스를 완성하는 leg

forfeit는 새 멱등성·락 메커니즘을 만들지 않는다. [007](007.md)이 세운 `runIdempotent`(fast path / write path / race-loser) + `writePair` primitive를 그대로 재사용하고, leg 매핑과 reason만 추가한다. [007](007.md)의 표기(`DEBIT 목적지 ← CREDIT 출발지`)로 정산 4결과를 나란히 두면 빈칸이 보인다.

```
win  : DEBIT user.AVAILABLE ← CREDIT user.LOCKED      (BET_REFUND  · 스테이크 반환)
     + DEBIT user.AVAILABLE ← CREDIT HOUSE.AVAILABLE  (BET_PAYOUT  · 수익 지급)
push : DEBIT user.AVAILABLE ← CREDIT user.LOCKED      (BET_REFUND)
void : DEBIT user.AVAILABLE ← CREDIT user.LOCKED      (BET_REFUND)
lose : DEBIT HOUSE.AVAILABLE ← CREDIT user.LOCKED     (BET_FORFEIT)   ← 이 커밋
```

`WalletService.forfeit`는 `runIdempotent`로 감싸 사용자 계좌에 비관적 락을 걸고, `account.forfeitLocked(amount, now)`로 도메인 잔고를 바꾼 뒤 `writePair`로 HOUSE-debit / user-LOCKED-credit 쌍을 쓴다. 다른 연산과 달리 **business-rejection 분기가 없다** — locked bucket이 접수 시 이미 그 스테이크를 들고 있어 정상 흐름에서 잔고 부족이 나오지 않는다. 잘못된 호출(locked보다 큰 금액)만 도메인 mutator가 `InsufficientBalanceException`으로 막는다.

```java
public WalletOperationResult forfeit(ForfeitCommand cmd) {
  return runIdempotent(cmd.idempotencyKey(), cmd.userId(), cmd.amount(), LedgerReason.BET_FORFEIT,
      () -> {
        Account account = lockAccount(cmd.userId(), cmd.amount());
        Instant now = clock.instant();
        account.forfeitLocked(cmd.amount(), now);
        return writePair(
            new TransferLeg(SystemAccountIds.HOUSE, BalanceBucket.AVAILABLE),
            new TransferLeg(cmd.userId(), BalanceBucket.LOCKED),
            cmd.amount(), LedgerReason.BET_FORFEIT, cmd.idempotencyKey(), cmd.userId(), now);
      });
}
```

### V3 마이그레이션 — CHECK 제약 확장

V1 스키마의 `ledger_entry_reason_valid` CHECK는 `DEPOSIT / WITHDRAW / BET_DEBIT / BET_PAYOUT / BET_REFUND` 다섯만 허용했다. 새 `BET_FORFEIT` 행은 이 제약에 걸려 거부되므로, 제약을 DROP한 뒤 reason 하나를 더해 다시 ADD한다. 기존 행은 모두 원래 다섯 reason이라 재검증은 no-op이다.

```sql
ALTER TABLE ledger_entry DROP CONSTRAINT ledger_entry_reason_valid;
ALTER TABLE ledger_entry ADD CONSTRAINT ledger_entry_reason_valid CHECK (reason IN (
    'DEPOSIT','WITHDRAW','BET_DEBIT','BET_PAYOUT','BET_REFUND','BET_FORFEIT'));
```

### 멱등성 — settlement의 재전달에 안전

settlement은 LOST 베팅마다 `settle:forfeit:{betId}` 키로 이 endpoint를 부른다. Kafka 재전달이나 admin replay로 같은 LOST가 두 번 와도, [007](007.md)의 멱등성 3층이 두 번째 호출을 원래 결과로 흡수해 **double-capture(스테이크 이중 회수)가 일어나지 않는다**. forfeit는 새 키를 쓸 뿐, 멱등성 보장 자체는 기존 primitive가 한다.

## 결과

- LOST 정산이 wallet에 실제 leg를 갖는다. 진 베팅의 스테이크가 user의 locked에서 빠져 HOUSE로 들어가고, 복식부기 합 = 0 invariant는 유지된다(HOUSE-debit이 user-LOCKED-credit과 상쇄).
- 통합 테스트: deposit 100k → debit 30k(locked 30k) → forfeit 30k → **locked 0, available 70k(불변), ledger net 0**. over-locked forfeit는 `InsufficientBalanceException`으로 거절되고 분개를 한 줄도 쓰지 않는다.
- 전체 스택 e2e 라이브 검증: 한 이벤트 3 selection에 single 3건(1 WON / 2 LOST) → wallet ledger에 HOUSE `BET_FORFEIT` DEBIT 2건, user locked 0, net 0.

## 요약

- LOST 정산의 빠진 wallet leg를 채웠다: `BET_FORFEIT`로 스테이크를 user.LOCKED → HOUSE.AVAILABLE로 capture.
- 새 메커니즘 0 — [007](007.md)의 `runIdempotent` + `writePair`를 재사용하고 leg 매핑·reason만 추가했다.
- V3 마이그레이션이 CHECK 제약에 reason 하나를 더한다.
- 정산 4결과(win / push / void / lose)가 모두 wallet 연산을 갖게 됐다.
- **설계 시점에 만들어 둔 primitive(`forfeitLocked` + HOUSE 계좌)가 통합 단계에서 비로소 배선됐다** — "연결 안 됨"은 단위 테스트로 드러나지 않고 전체 스택 정산에서 드러났다.

## 다음 작업

- wallet 측 follow-up 없음. 정산 매트릭스는 이제 완결이다.
- forfeit는 Kafka 이벤트를 발행하지 않는다(settlement이 HTTP로 직접 구동). 운영에서 forfeit 이벤트 구독이 필요해지면 deposit/withdraw와 같은 결의 shared-protocol 이벤트 추가가 필요하다(change-cost의 입출금 이벤트 항목과 동형).

## 핵심 확인

- 테스트: `WalletServiceIntegrationTest.forfeit_captures_locked_stake_to_house`(locked→house, net 0), `forfeit_above_locked_rejects_and_writes_no_entry`(거절·무기록), `WalletControllerTest.forfeit_ok`(200 + reason BET_FORFEIT).
- forfeit에는 business-rejection 분기가 없다 — locked가 접수 시 이미 스테이크를 들고 있기 때문이다.

## 기억/설명 Level

### L3 - 백지 설명 대상

- 정산 4결과를 wallet leg로 매핑하는 전체 그림: win = REFUND + PAYOUT, push/void = REFUND, **lose = FORFEIT(locked→house)**. forfeit가 빠지면 진 스테이크가 user.locked에 영영 묶인다는 결함, 그리고 도메인 primitive(`forfeitLocked` + HOUSE 계좌)는 있었지만 LOST 경로가 미배선이었다는 점.

### L2 - 코드/문서 기반 설명 대상

- forfeit가 [007](007.md)의 `runIdempotent` / `writePair`를 재사용하고 business-rejection 분기 없이 leg 매핑만 더하는 구조, `settle:forfeit:{betId}` 멱등 키로 재전달을 흡수하는 점.
- V3 CHECK 제약 확장(DROP 후 reason 추가 재ADD, 기존 행 재검증 no-op).

### L1 - 읽으면 충분

- `BET_FORFEIT` enum 추가, `POST /transactions/forfeit` endpoint 시그니처.
