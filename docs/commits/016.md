# after-commit InvariantChecker

- 해시: `809292f` feat(integrity): add after-commit InvariantChecker for ledger pair integrity
- 커밋된 연산마다 분개 쌍의 네 가지 무결성을 검증하는 after-commit 리스너를 둔다

## 개요

`InvariantChecker`는 `OperationCommitted` 도메인 이벤트를 `@TransactionalEventListener(AFTER_COMMIT)`로 구독해, 성공적으로 커밋된 모든 wallet 연산에 대해 네 가지를 검증한다. DB unique 제약과 `pair()` 팩토리가 나쁜 쌍이 애초에 들어오지 못하게 막는 1차 방어이고, 이 리스너는 조용한 위반(직접 INSERT, 버그 있는 마이그레이션, 운영자의 fix-it SQL)이 눈에 안 띄게 지나가지 못하게 하는 2차 장치다.

## 작업 순서

1. `OperationCommitted` 이벤트 record를 둔다.
2. `WalletService.writePair`가 쌍 저장 직후 이벤트를 publish한다.
3. `InvariantChecker`가 AFTER_COMMIT에 네 가지를 검증한다.
4. Micrometer 카운터로 위반/검사 횟수를 노출한다.

## 작업 내역

### 검증 네 가지

1. operation_group_id 아래 정확히 두 entry(차변 1, 대변 1).
2. 차변 1 + 대변 1(둘 다 차변이거나 둘 다 대변이면 안 됨).
3. 차변 금액 − 대변 금액 = 0.
4. 두 entry가 같은 통화.

```java
@TransactionalEventListener(phase = TransactionPhase.AFTER_COMMIT)
public void verify(OperationCommitted event) {
  checks.increment();
  List<LedgerEntry> entries = ledgerRepo.findByOperationGroupId(event.operationGroupId());
  if (entries.size() != EXPECTED_PAIR_SIZE) { violations.increment(); log.error(...); return; }
  // side 분리, net == 0, 통화 일치 검사
}
```

### AFTER_COMMIT 타이밍

`WalletService`는 `writePair()`에서 쌍을 저장한 직후 `OperationCommitted`를 publish한다. 이벤트는 감싸는 트랜잭션이 커밋될 때까지 버스에 머문다. AFTER_COMMIT phase 덕에 롤백된 시도(잔고 부족, 제약 race-loser, 일반 DataIntegrityViolation)에는 리스너가 fire하지 않는다.

위반을 발견해도 커밋 후라 되돌릴 수 없다. ERROR 로그와 `wallet.invariant.violations` Micrometer 카운터 증가로 운영자가 기존 Prometheus 알람 체인으로 페이징된다(ADR-0007). `wallet.invariant.checks` 카운터로 위반율을 계산 가능하게 한다.

### repository 집계 메서드

리스너와 다음 reconciliation 커밋을 위해 세 메서드가 추가됐다. `findByOperationGroupId`(리스너가 알림받은 쌍 조회), `netSumAll`(시스템 전체 차변−대변, 정상 시 0), `netByAccountAndBucket`(계좌·bucket별 차변−대변).

## 결과

매 연산 후 분개 쌍의 무결성이 자동 검증된다. 트랜잭션 시점 방어(제약/팩토리)가 진짜 예방이고 이 리스너는 조용한 위반을 불가능하게 만드는 안전망이다.

## 요약

- AFTER_COMMIT 리스너가 커밋된 연산마다 네 무결성(쌍 크기/side 분리/net=0/통화)을 검증한다.
- 롤백된 시도에는 fire하지 않는다.
- 위반은 되돌릴 수 없으므로 ERROR 로그 + Micrometer 카운터로 운영자에게 알린다.

## 다음 작업

매일 도는 전체 ledger reconciliation 배치를 추가한다.

## 핵심 확인

- AFTER_COMMIT은 롤백된 트랜잭션에 fire하지 않는다.
- 리스너는 예방이 아니라 안전망이다. 예방은 DB 제약과 `pair()` 팩토리가 한다.
- `wallet.invariant.violations`가 운영 알람 트리거다.

## 기억/설명 Level

### L3 - 백지 설명 대상

- after-commit invariant 검증이 트랜잭션 시점 방어(제약/팩토리)와 어떻게 역할을 나누는지, 그리고 위반을 되돌릴 수 없어 알람으로 대응하는 운영 모델
