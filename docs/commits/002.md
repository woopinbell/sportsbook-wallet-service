# LedgerEntry 분개장 도입 (double-entry)

- 해시: `62d6247` feat(domain): introduce LedgerEntry append-only journal rows
- 모든 자금 이동을 차변/대변 쌍으로 기록하는 append-only 분개 entry와 멱등성 제약 모델을 정의한다

## 개요

`LedgerEntry`는 double-entry ledger의 불변 building block이다. 모든 자금 이동은 `LedgerEntry.pair()`가 만드는 차변(debit)/대변(credit) 쌍으로 기록된다. 두 entry는 같은 `operation_group_id`와 `idempotency_key`를 공유하고 `side`로만 갈린다. 차변은 목적지 (계좌, bucket)에, 대변은 출발지에 붙는다.

이 repo가 쓰는 규약은 차변 = bucket 증가, 대변 = bucket 감소다. 시스템 전체에서 차변 합에서 대변 합을 뺀 값은 항상 0이다. 이 invariant가 이후 after-commit listener와 일일 reconciliation이 검증하는 대상이다.

## 작업 순서

1. side / bucket / reason / 시스템 계좌 ID를 작은 enum과 상수로 박는다.
2. UUID v7 생성기를 둔다(시간 정렬 PK).
3. `LedgerEntry` entity와 `pair()` 팩토리를 만든다.
4. 멱등성 제약을 `UNIQUE(idempotency_key, side)`로 설계한다.

## 작업 내역

### 멱등성의 matched-pair 충돌

ADR-0005는 `idempotency_key`에 unique 제약을 걸라고 한다. 그런데 double-entry는 한 연산이 같은 키를 가진 두 행을 만든다. 단순 `UNIQUE(idempotency_key)`는 쌍의 두 번째 행을 거부한다. 튜플에 `side`를 넣어 해결한다.

- `UNIQUE(idempotency_key, side)`: 키당 차변 1개 + 대변 1개. 같은 키 재시도는 아무 행도 추가하지 못한다.
- `UNIQUE(operation_group_id, side)`: 내부 ID에 대한 방어적 2차 제약.

이 충돌은 ADR을 읽을 때는 안 보이고 double-entry 구조와 만나서야 드러난다. 멱등성 자체를 DB 제약으로 강제하는 것이 핵심이고, 애플리케이션 체크는 가속일 뿐이다.

### `LedgerEntry.pair()`

```java
public static Pair pair(
    TransferLeg destination,
    TransferLeg source,
    Money amount,
    LedgerReason reason,
    IdempotencyKey idempotencyKey,
    UUID operationGroupId,
    Instant now) {
  Objects.requireNonNull(destination, "destination");
  Objects.requireNonNull(source, "source");
  LedgerEntry debit =
      new LedgerEntry(
          UuidV7.generate(),
          destination.accountId(),
          destination.bucket(),
          LedgerSide.DEBIT,
          amount,
          reason,
          idempotencyKey,
          operationGroupId,
          now);
  LedgerEntry credit =
      new LedgerEntry(
          UuidV7.generate(),
          source.accountId(),
          source.bucket(),
          LedgerSide.CREDIT,
          amount,
          reason,
          idempotencyKey,
          operationGroupId,
          now);
  return new Pair(debit, credit);
}
```

`TransferLeg`와 `Pair`는 record로 두어 (계좌, bucket) 참조와 (차변, 대변) 결과를 타입으로 묶었다. Java 17 record는 C++의 구조체 비슷하지만 불변이고 `equals`/`hashCode`가 자동 생성된다.

### enum과 시스템 계좌

세 enum은 각자 파일이다. 아래는 각 파일의 선언부다.

```java
public enum BalanceBucket {
  /** Spendable funds. Hits this bucket on deposit, withdraw, payout, refund. */
  AVAILABLE,
  /** Held against open bet exposure. Filled on stake, drained on settlement. */
  LOCKED
}

public enum LedgerSide {
  /** Balance on the target bucket goes up. */
  DEBIT,
  /** Balance on the target bucket goes down. */
  CREDIT
}

public enum LedgerReason {
  /** External money flowing into the user's account. */
  DEPOSIT,
  /** External money leaving the user's account. */
  WITHDRAW,
  /** Stake moving from the user's available bucket to their locked bucket on bet acceptance. */
  BET_DEBIT,
  /** Profit paid out by the house when a bet wins. */
  BET_PAYOUT,
  /** Stake returned from locked back to available (void / push / cancelled event). */
  BET_REFUND
}
```

`SystemAccountIds.HOUSE`와 `EXTERNAL_PAYMENT`는 고정 UUID다. 외부 세계가 차변/대변을 걸 상대를 주어서, 외부 상태를 모델링하지 않고도 합 = 0 invariant가 성립하게 한다. 둘 다 알아보기 쉬운 all-zero 패턴에 유효한 v7 version/variant 비트를 박았다.

### `UuidV7`

RFC 9562 §5.7 생성기다(48비트 unix-ms + 4 version + 12 random + 2 variant + 62 random). 시간 정렬 prefix 덕분에 PK가 B-tree에 친화적인 순차 삽입 패턴을 유지한다(ADR-0003). 각 서비스가 UUID 라이브러리를 또 끌어오는 대신 자체 생성기를 둔다.

## 결과

이후 service의 모든 연산은 `pair()` 한 곳을 거쳐 분개를 만든다. 멱등성 제약이 DB 레벨에 박혔으므로, 같은 키 동시 요청 중 하나만 쌍을 쓰고 나머지는 제약 위반으로 막힌다.

## 요약

- 차변 = bucket 증가, 대변 = bucket 감소. 시스템 합은 항상 0.
- `UNIQUE(idempotency_key, side)`가 matched-pair와 멱등성을 동시에 성립시킨다.
- `pair()`가 모든 자금 이동의 단일 통로다.
- UUID v7로 PK를 시간 정렬해 인덱스 삽입을 순차화한다.

## 핵심 확인

- 단순 `UNIQUE(idempotency_key)`는 double-entry 쌍의 두 번째 행을 거부한다. `side`를 튜플에 넣어야 한다.
- `pair()`는 차변/대변에 같은 `operation_group_id`와 `idempotency_key`를 부여한다.
- 시스템 계좌(HOUSE / EXTERNAL_PAYMENT)는 account 테이블에 행이 없는 고정 UUID다.

## 기억/설명 Level

### L3 - 백지 설명 대상

- double-entry의 차변/대변 규약과 시스템 합 = 0 invariant, 그리고 그것이 이후 두 정합성 장치의 검증 대상이 되는 구조
- 멱등성을 `UNIQUE(idempotency_key, side)`로 강제하면서 matched-pair 두 행을 허용하는 제약 설계

### L2 - 코드/문서 기반 설명 대상

- `pair()` 팩토리가 (목적지, 출발지) leg로 차변/대변을 만드는 흐름
- `SystemAccountIds`가 외부 상태 없이 합 = 0을 성립시키는 역할
- UUID v7의 시간 정렬 prefix가 B-tree 삽입에 주는 이점
