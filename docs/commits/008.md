# WalletService 통합 테스트 (Testcontainers)

- 해시: `e8ac469` test(service): cover WalletService with Testcontainers PostgreSQL + Redis
- 실제 PostgreSQL + Redis 컨테이너 위에서 happy path / 거부 / 멱등성을 덮고 100건 동시 debit race를 증명한다

## 개요

Testcontainers는 테스트 실행 중 실제 Docker 컨테이너(여기서는 `postgres:16-alpine`, `redis:7-alpine`)를 띄우는 라이브러리다. mock이나 in-memory H2 대신 진짜 PostgreSQL을 쓰므로, `SELECT FOR UPDATE`와 unique 제약 같은 DB 고유 동작이 테스트에서도 그대로 검증된다. C/C++에는 직접 대응이 없는 도구로, 통합 테스트의 신뢰도를 실제 런타임에 맞춘다.

여덟 테스트가 세 nested 그룹으로 나뉜다. Kafka auto-configuration은 이 슬라이스에서 제외한다. outbox 테스트(012)가 Kafka 컨테이너를 따로 가져올 것이고, 여기서 띄우면 새 경로 검증 없이 시작 시간만 10초 넘게 늘기 때문이다.

## 작업 순서

1. `@Testcontainers`로 PostgreSQL + Redis 컨테이너를 띄운다.
2. `@DynamicPropertySource`로 컨테이너 접속 정보를 Spring 프로퍼티에 주입한다.
3. happy path 3 / 거부 2 / 멱등성 3 테스트를 짠다.
4. 100건 동시 debit을 `CompletableFuture`로 쏴 race를 증명한다.

## 작업 내역

### 컨테이너 배선

```java
@Container
static PostgreSQLContainer<?> postgres = new PostgreSQLContainer<>("postgres:16-alpine");

@Container
static GenericContainer<?> redis =
    new GenericContainer<>("redis:7-alpine").withExposedPorts(6379);

@DynamicPropertySource
static void registerProperties(DynamicPropertyRegistry registry) {
  registry.add("spring.datasource.url", postgres::getJdbcUrl);
  registry.add("spring.datasource.username", postgres::getUsername);
  registry.add("spring.datasource.password", postgres::getPassword);
  registry.add("spring.data.redis.host", redis::getHost);
  registry.add("spring.data.redis.port", () -> redis.getMappedPort(6379));
}
```

`application-test.yml`은 datasource URL을 비워둔다. 각 통합 테스트가 자기 `@DynamicPropertySource`로 컨테이너를 주입할 것을 강제하고, 빠뜨리면 오래된 로컬 PostgreSQL에 조용히 붙는 대신 `NoSuchBeanDefinitionException`으로 드러나게 한다.

### 세 그룹

- **HappyPath(3)**: deposit→debit→credit이 계좌를 원래 available로 되돌리고 locked를 0으로 만든다. 전체 테이블의 차변−대변이 0으로 수렴한다. withdraw가 EXTERNAL_PAYMENT 상대로 쌍을 쓴다. HOUSE_POOL credit이 수익을 지급하되 합 = 0이 유지된다.
- **Rejection(2)**: available 초과 debit이 `InsufficientBalanceException`을 던지고 ledger에는 원래 deposit 쌍만 남는다. KRW 계좌에 USD deposit이 `CurrencyMismatchException`.
- **Idempotent(3)**: 같은 키 두 번 호출이 두 번째에 같은 `operation_group_id`를 돌려주고 쌍을 더 만들지 않는다. 같은 키 다른 금액이 `IdempotencyConflictException`.

### 100건 동시 debit

```java
int parallelism = 100;
ExecutorService exec = Executors.newFixedThreadPool(20);
CountDownLatch start = new CountDownLatch(1);
List<CompletableFuture<WalletOperationResult>> futures = new ArrayList<>();

for (int i = 0; i < parallelism; i++) {
  futures.add(
      CompletableFuture.supplyAsync(
          () -> {
            try {
              start.await();
            } catch (InterruptedException e) {
              Thread.currentThread().interrupt();
              throw new IllegalStateException(e);
            }
            return walletService.debit(new DebitCommand(userId, krw(1_000), raceKey));
          },
          exec));
}
start.countDown();
```

100개 future가 `start.await()`로 대기하다 `countDown()`과 함께 같은 키로 동시에 debit한다. 이어지는 단언은 모든 결과의 `operation_group_id`가 하나로 수렴하는지, 그 키의 ledger pair가 정확히 1개(2행)인지, 전체 합이 0인지를 본다.

race-loser들이 내는 unique 제약 위반은 `WalletService`의 재시도 handler가 흡수한다. 테스트 로그에 PostgreSQL warning으로 보이지만 테스트 실패로는 올라오지 않는다. 이것이 부하 테스트(019)가 더 큰 규모로 다시 돌릴 Level 3 정합성 증명이다.

## 결과

여덟 테스트 모두 통과했다. 동시 debit이 정확히 하나의 분개 쌍만 만들고 잔고가 일관되게 유지됨을, 실제 PostgreSQL의 락과 제약 위에서 확인했다.

## 요약

- Testcontainers로 실제 PostgreSQL/Redis를 띄워 DB 고유 동작을 검증한다.
- `@DynamicPropertySource`로 컨테이너 접속 정보를 주입하고, 빈 기본값으로 누락을 드러낸다.
- 100건 동시 debit이 단일 쌍으로 수렴함을 `CompletableFuture` race로 증명한다.
