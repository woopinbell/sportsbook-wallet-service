# LedgerEntryRepository 읽기 메서드

- 해시: `68e3f49` feat(repository): add LedgerEntryRepository read methods
- append-only 분개장의 멱등성 조회와 계좌별 시간순 스캔을 파생 쿼리로 노출한다

## 개요

`LedgerEntryRepository`는 append-only 분개장의 읽기 경로만 노출한다. Spring Data의 파생 쿼리(derived query)는 메서드 이름을 파싱해 쿼리를 생성한다. `findByIdempotencyKey`라는 이름만으로 `WHERE idempotency_key = ?`가 만들어진다. 이름 규칙을 알면 SQL을 안 적어도 되고, 모르면 이름이 곧 쿼리라는 점이 헷갈릴 수 있다.

## 작업 순서

1. `JpaRepository`를 상속한다.
2. 멱등성 조회 두 개(`findByIdempotencyKey`, `existsByIdempotencyKey`)를 둔다.
3. 계좌별 시간순 스캔(`findByAccountIdOrderByCreatedAtAsc`)을 둔다.

## 작업 내역

### `LedgerEntryRepository.java`

```java
public interface LedgerEntryRepository extends JpaRepository<LedgerEntry, UUID> {

  /** Returns both entries of a matched pair, or empty if the key has not been processed. */
  List<LedgerEntry> findByIdempotencyKey(String idempotencyKey);

  /** Cheap existence probe — used by the Redis-miss fast path before falling through to insert. */
  boolean existsByIdempotencyKey(String idempotencyKey);

  /** All entries for a single account, oldest first. */
  List<LedgerEntry> findByAccountIdOrderByCreatedAtAsc(UUID accountId);
}
```

세 파생 쿼리가 각자 다른 경로를 받친다.

- `findByIdempotencyKey`: 키에 대한 차변/대변 쌍을 돌려준다. 재시도 복구 handler가 이미 이긴 쪽의 응답을 재구성할 때 쓴다.
- `existsByIdempotencyKey`: Redis 미스 시 INSERT 시도 전 존재 확인용 가벼운 probe.
- `findByAccountIdOrderByCreatedAtAsc`: 계좌별 시간순 스캔. after-commit invariant listener와 일일 reconciliation이 쓴다.

인터페이스에 변경(mutating) 메서드는 두지 않는다. 분개장은 append-only이고, 프로덕션 코드는 `saveAll(pair)`로 새 entry 삽입만 한다. Spring Data가 상속으로 주는 `save`/`delete`는 fixture가 필요한 테스트만 쓴다.

## 결과

service 레이어가 멱등성 fast path와 분개 audit에 필요한 읽기를 모두 갖췄다. 이후 reconciliation 커밋에서 집계 쿼리(`netSumAll`, `netByAccountAndBucket`)가 이 인터페이스에 추가된다.

## 요약

- 파생 쿼리 세 개로 멱등성 조회와 계좌 시간순 스캔을 노출한다.
- 변경 메서드는 두지 않는다. 분개장은 append-only다.
