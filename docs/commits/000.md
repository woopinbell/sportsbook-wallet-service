# wallet-service 골격 초기화

- 해시: `7f54434` chore(project): initialize wallet-service layout
- Spring Boot 3.2 / Java 17 / Maven 프로젝트 골격과 의존성, 빌드 플러그인, 빈 진입점을 세운다

## 개요

이 커밋은 이후 모든 코드가 올라갈 빌드 골격을 만든다. double-entry ledger를 담을 서비스이므로, 시작부터 영속성(PostgreSQL + Flyway), 메시징(Kafka + Avro), 관측성(JSON 로깅 + OTel + Prometheus), 테스트(Testcontainers) 의존을 모두 깔아둔다.

C/C++의 Makefile과 다른 점은 Maven이 의존성 해석과 버전 정렬을 BOM(Bill of Materials)으로 한다는 점이다. `spring-boot-starter-parent`를 parent로 상속하면 수십 개 라이브러리의 버전을 직접 적지 않아도 Spring Boot가 검증한 조합으로 맞춰준다. 직접 버전을 박는 곳은 BOM이 모르는 것(shared-protocol, Avro, Testcontainers)뿐이다.

## 작업 순서

1. `spring-boot-starter-parent` 3.2.11을 parent로 잡는다.
2. shared-protocol(mavenLocal) + starter 묶음 + PostgreSQL/Flyway + Kafka/Avro + 관측성 + Lombok + 테스트 스택을 의존으로 추가한다.
3. Spotless(google-java-format)와 Checkstyle을 verify phase에 바인딩한다.
4. 빈 `WalletServiceApplication` 진입점과 최소 `application.yml` / `application-test.yml`을 둔다.
5. `.gitignore`에 `CLAUDE.md`와 빌드 산출물을 박는다.
6. `mvn verify`로 골격이 컴파일되고 shared-protocol이 mavenLocal에서 해석되는지 확인한다.

## 작업 내역

### `pom.xml`

shared-protocol은 mavenLocal에서 받는다. wallet은 그 라이브러리가 생성한 Avro 클래스와 같은 Avro 1.12.x 런타임을 써야 바이너리 호환이 된다(ADR-0014).

```xml
<parent>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-parent</artifactId>
    <version>3.2.11</version>
    <relativePath/>
</parent>
```

Lombok은 `annotationProcessorPaths` 안에서 버전을 직접 박았다. 이 블록은 Spring Boot BOM을 참조하지 않기 때문에 버전을 비워두면 빌드가 "version can neither be null"로 깨진다. C/C++에는 없는 annotation processor라는 컴파일 단계 코드 생성기를 Maven이 별도 classpath로 다루는 데서 오는 함정이다.

```xml
<path>
    <groupId>org.projectlombok</groupId>
    <artifactId>lombok</artifactId>
    <version>${lombok.version}</version>
</path>
```

### `WalletServiceApplication.java`

진입점은 비어 있지만 `@EnableScheduling`을 클래스 레벨에 올려둔다. 이후 outbox publisher와 일일 reconciliation 배치가 `@Scheduled`를 선언할 때 추가 설정 없이 동작하게 하기 위함이다.

```java
@SuppressWarnings("checkstyle:HideUtilityClassConstructor")
@SpringBootApplication
@EnableScheduling
public class WalletServiceApplication {

  public static void main(String[] args) {
    SpringApplication.run(WalletServiceApplication.class, args);
  }
}
```

`@SuppressWarnings`는 Checkstyle이 이 `@Configuration` 클래스를 유틸리티 클래스로 오인해 private 생성자를 요구하는 것을 막는다. Spring이 이 클래스를 bean으로 인스턴스화하므로 private 생성자를 두면 안 된다.

### 설정과 `.gitignore`

`application.yml`은 Boot 컨텍스트가 뜨는 데 필요한 최소(datasource, jpa, kafka, redis, actuator)만 둔다. 테스트 프로파일은 datasource를 Testcontainers의 `jdbc:tc:postgresql` URL로 가리켜 외부 PostgreSQL 없이 통합 테스트가 돌게 한다. `.gitignore`는 `CLAUDE.md`와 `.claude/`를 추적에서 뺀다. AI 세션 컨텍스트 파일이지 프로젝트 산출물이 아니기 때문이다.

## 결과

`mvn verify -DskipTests`가 Spotless와 Checkstyle을 통과하고, `mvn dependency:tree -Dincludes=com.sportsbook:shared-protocol`이 mavenLocal에서 jar를 해석했다. 이후 커밋은 의존성을 다시 건드리지 않고 도메인 코드부터 올린다.

## 요약

- BOM 상속으로 라이브러리 버전 정렬을 Spring Boot에 위임하고, BOM 밖 의존(shared-protocol/Avro/Testcontainers)만 직접 버전을 박았다.
- Spotless + Checkstyle을 verify에 바인딩해 `mvn verify`가 포맷과 의미 규칙 회귀를 잡는다.
- `@EnableScheduling`을 미리 켜 이후 스케줄러 기능이 배선 없이 붙는다.

## 핵심 확인

- shared-protocol은 mavenLocal 의존이므로 wallet 빌드 전에 shared-protocol에서 `mvn install`이 선행돼야 한다.
- Lombok 버전을 `annotationProcessorPaths`에 직접 박지 않으면 빌드가 깨진다.
- `CLAUDE.md`는 어떤 커밋에도 들어가지 않는다(`.gitignore` 박제).
