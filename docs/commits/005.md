# AccountRepository 비관적 락 조회

- 해시: `6348505` feat(repository): add AccountRepository with PESSIMISTIC_WRITE lookup
- 잔고를 바꾸는 모든 경로가 거치는 `SELECT FOR UPDATE` 조회를 repository에 노출한다

## 개요

Spring Data JPA는 인터페이스만 선언하면 구현을 런타임에 생성하는 repository 추상화다. C/C++의 DAO를 손으로 짜는 것과 달리, 메서드 시그니처와 어노테이션만으로 쿼리가 만들어진다. `AccountRepository`는 두 가지 읽기 경로를 구분해 노출한다. 락 없는 조회와 락 거는 조회다.

## 작업 순서

1. `JpaRepository<Account, UUID>`를 상속해 기본 CRUD를 얻는다.
2. `findByUserIdForUpdate`에 `@Lock(PESSIMISTIC_WRITE)`를 걸어 `SELECT FOR UPDATE`로 만든다.
3. 동반 예외 `AccountNotFoundException`을 둔다.

## 작업 내역

### `AccountRepository.java`

```java
public interface AccountRepository extends JpaRepository<Account, UUID> {
  @Lock(LockModeType.PESSIMISTIC_WRITE)
  @Query("select a from Account a where a.userId = :userId")
  Optional<Account> findByUserIdForUpdate(@Param("userId") UUID userId);
}
```

두 경로의 의미가 다르다.

- `findById(userId)`: JpaRepository 상속, 락 없음. 잔고 조회 endpoint와 after-commit invariant listener가 쓴다.
- `findByUserIdForUpdate(userId)`: 행에 `SELECT ... FOR UPDATE`를 건다. 잔고를 바꾸는 연산만 이 경로를 쓴다. ADR-0005의 비관적 락 정책이다. 사용자 계좌 행을 락으로 잡은 채 같은 트랜잭션 안에서 ledger entry가 들어가고, 이것이 같은 사용자에 대한 동시 debit을 직렬화하는 메커니즘이다.

`@Query`와 `@Lock`을 명시적으로 조합한 것은 Spring Data의 메서드 이름 파생(keyword derivation)에 의존하는 것보다 메서드 이름이 호출부 의도("행을 락 걸고 가져온다")로 읽히기 때문이다.

### `AccountNotFoundException`

`com.sportsbook.wallet.domain.error`에 둔다. 기존 `InsufficientBalanceException`, `CurrencyMismatchException`과 한자리에 모아, service가 문자열 검사 없이 던지고 이후 ControllerAdvice가 셋을 RFC 7807로 일관되게 매핑한다.

## 결과

잔고 변경 경로는 이제 `findByUserIdForUpdate`로 행을 잡는다. PostgreSQL의 row lock은 트랜잭션 커밋까지 유지되므로, 같은 사용자에 대한 두 번째 동시 트랜잭션은 첫 번째가 끝날 때까지 그 행에서 블록된다. C/C++의 mutex와 다른 점은 락의 단위가 코드 블록이 아니라 DB row이고 해제가 커밋에 묶인다는 점이다.

## 요약

- 락 없는 `findById`와 락 거는 `findByUserIdForUpdate` 두 경로를 구분한다.
- `@Lock(PESSIMISTIC_WRITE)`가 `SELECT FOR UPDATE`로 번역돼 동시 debit을 직렬화한다.
- 도메인 예외 셋을 한 패키지에 모아 일관된 오류 매핑을 준비한다.

## 핵심 확인

- 잔고 변경은 반드시 `findByUserIdForUpdate`를 거친다. 락 없는 조회로 바꾸면 동시 debit이 직렬화되지 않는다.
- row lock은 트랜잭션 커밋까지 유지된다.

## 기억/설명 Level

### L3 - 백지 설명 대상

- `@Lock(PESSIMISTIC_WRITE)`가 사용자 계좌 행을 잡아 동시 debit을 직렬화하는 메커니즘과, 락 해제가 트랜잭션 커밋에 묶인다는 점

### L2 - 코드/문서 기반 설명 대상

- 락 없는 조회와 락 거는 조회를 메서드로 분리해 호출부 의도를 드러내는 설계
- 도메인 예외 셋을 한 패키지에 모아 ControllerAdvice 일관 매핑을 준비하는 구성
