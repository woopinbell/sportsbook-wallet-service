# WalletService 멱등 전송 primitive

- 해시: `92c6564` feat(service): introduce WalletService with idempotent transfer primitive
- deposit / withdraw / debit / credit 네 연산이 공유하는 멱등 전송 primitive를 세 층의 멱등성 보장과 함께 만든다

## 개요

`WalletService`는 쓰기 측 핵심 애플리케이션 서비스다. V1의 네 연산이 하나의 전송 primitive를 공유하고, 그 primitive가 멱등성 계약(idempotency contract)을 끝까지 책임진다. 이 커밋이 이 repo에서 가장 밀도 높은 결정 묶음이다. 비관적 락, 멱등성 3층, 트랜잭션 경계 선택이 한자리에 모인다.

## 작업 순서

1. Redis fast path 캐시(`IdempotencyCache`)를 둔다.
2. `runIdempotent` 골격에 fast path / write path / race-loser path 세 층을 짠다.
3. 네 연산을 transfer leg 매핑으로 primitive 위에 올린다.
4. 트랜잭션 경계를 `TransactionTemplate`(프로그래밍 방식)으로 잡는다.
5. `Clock`을 주입해 시각을 테스트에서 고정 가능하게 한다.

## 작업 내역

### 멱등성 3층 (ADR-0005)

1. **Fast path**: `IdempotencyCache`(Redis `StringRedisTemplate`, 24h TTL, 키 prefix `idempotency:wallet:`)가 키를 본 적 있으면 원래 전송의 `operation_group_id`를 돌려준다. 그 그룹의 분개 쌍을 DB에서 가져와 원래 호출이 돌려준 것과 같은 `WalletOperationResult`로 재구성한다.

2. **Write path**: `runIdempotent()`가 `TransactionTemplate`으로 트랜잭션을 열고, `findByUserIdForUpdate`로 사용자 계좌에 `SELECT FOR UPDATE`를 건 뒤, 도메인 mutator로 잔고를 바꾸고, `pair()`로 차변/대변을 삽입한다. 커밋 후 `operation_group_id`를 Redis에 캐싱한다.

3. **Race-loser path**: 같은 키의 동시 요청 중 하나가 이겨서 `UNIQUE(idempotency_key, side)` 제약이 터지면, 진 트랜잭션은 롤백되고 catch가 이긴 쪽 쌍을 새 트랜잭션에서 다시 읽는다. 재시도 payload(userId, amount, reason)를 원본과 대조해 불일치면 `IdempotencyConflictException` → HTTP 409, 동일하면 같은 결과를 돌려준다.

```java
private WalletOperationResult runIdempotent(IdempotencyKey key, UUID userId, Money amount,
    LedgerReason expectedReason, Supplier<WalletOperationResult> action) {
  Optional<UUID> cached = cache.lookupOperationGroup(key);
  if (cached.isPresent()) {
    Optional<WalletOperationResult> existing = loadExisting(key);
    if (existing.isPresent()) { validateRetryMatches(...); return existing.get(); }
  }
  try {
    WalletOperationResult result = writeTx.execute(status -> action.get());
    cache.markProcessed(key, result.operationGroupId());
    return result;
  } catch (DataIntegrityViolationException raceLost) {
    Optional<WalletOperationResult> winning = loadExisting(key);
    if (winning.isEmpty()) throw raceLost;
    validateRetryMatches(...);
    return winning.get();
  }
}
```

### 왜 `@Transactional`이 아니라 `TransactionTemplate`인가

race-loser path는 `DataIntegrityViolationException`을 잡은 뒤 이긴 쪽 쌍을 다시 읽어야 한다. 메서드 레벨 `@Transactional` 안에서 unique 위반이 나면 그 트랜잭션은 rollback-only로 표시되고, 같은 트랜잭션에서 추가 read를 하면 `TransactionSystemException`이 난다. 프로그래밍 방식 경계(`TransactionTemplate.execute`)는 catch를 실패한 write 트랜잭션 바깥에서 돌게 해 이 문제를 피한다. 선언적 트랜잭션을 "그냥 붙이면 되는 것"으로 보면 놓치는 함정이다.

### 연산별 leg 매핑

```
deposit:  DEBIT user.AVAILABLE  ← CREDIT EXTERNAL_PAYMENT.AVAILABLE
withdraw: DEBIT EXTERNAL_PAYMENT.AVAILABLE ← CREDIT user.AVAILABLE
debit:    DEBIT user.LOCKED     ← CREDIT user.AVAILABLE   (BET_DEBIT)
credit (USER_LOCKED): DEBIT user.AVAILABLE ← CREDIT user.LOCKED   (BET_REFUND)
credit (HOUSE_POOL):  DEBIT user.AVAILABLE ← CREDIT HOUSE.AVAILABLE (BET_PAYOUT)
```

각 연산은 적절한 (목적지, 출발지) leg로 `pair()`를 호출할 뿐이다. 멱등성과 락은 primitive가 처리한다.

`openAccount`는 유일하게 호출자 멱등성 키가 없는 연산이다. userId 자체가 자연 키다. PK 경합은 삽입 시 `DataIntegrityViolation`을 잡고 다시 읽어 처리한다.

### `Clock` 주입

`WalletInfrastructureConfig`가 `Clock`(UTC system clock)과 `TransactionTemplate`을 bean으로 wire한다. `WalletService`는 `Instant.now()`를 직접 부르지 않고 `Clock`에 의존해, 테스트가 JVM 상태를 건드리지 않고 시각을 고정할 수 있다.

## 결과

네 연산이 일관된 멱등성 보장을 받는다. 같은 키 재시도는 부수효과를 다시 실행하지 않고 원래 결과를 돌려주며, 동시 요청 중 하나만 분개 쌍을 쓴다. 이 동작은 이후 통합 테스트(008)와 부하 테스트(019)에서 실제로 증명된다.

## 요약

- fast path(Redis) / write path(비관적 락) / race-loser(DB 제약 복구) 세 층으로 멱등성을 보장한다.
- DB unique 제약이 강한 보장이고 Redis는 가속이다.
- `TransactionTemplate`을 쓰는 이유는 catch의 추가 read가 rollback-only 트랜잭션에 걸리지 않게 하기 위함이다.
- 네 연산은 leg 매핑만 다르고 primitive를 공유한다.

## 핵심 확인

- DB `UNIQUE(idempotency_key, side)`가 멱등성의 강한 보장이다. Redis는 fast path일 뿐이다.
- 메서드 레벨 `@Transactional`이었다면 race-loser의 재read가 `TransactionSystemException`을 냈을 것이다.
- `openAccount`만 호출자 키가 없고 userId가 자연 키다.

## 기억/설명 Level

### L3 - 백지 설명 대상

- 멱등성 3층(Redis fast path / 비관적 락 write path / DB 제약 race-loser 복구)과 각 층의 역할 분담
- `@Transactional` 대신 `TransactionTemplate`을 택한 이유(rollback-only 트랜잭션에서 추가 read 불가)
- 네 연산을 (목적지, 출발지) leg 매핑으로 단일 primitive 위에 올리는 구조

### L2 - 코드/문서 기반 설명 대상

- `runIdempotent` 골격의 분기 흐름과 `validateRetryMatches`의 payload 대조
- `openAccount`의 PK 경합 처리(삽입 실패 후 재read)
- `Clock` 주입으로 시각을 테스트 가능하게 만드는 구성
